37. How to optimize API?

Theory

API optimization focuses on reducing latency, improving throughput, and minimizing resource consumption. Key strategies include caching, compression, asynchronous operations, pagination, database query optimization, and proper HTTP status code usage. The goal is to serve more requests faster with less overhead while maintaining reliability.

How it works in .NET Core

Modern .NET Core APIs leverage several built-in mechanisms:

  • Response Caching Middleware: app.UseResponseCaching() with [ResponseCache] attribute
  • Output Caching (.NET 7+): More granular control with app.UseOutputCache()
  • Compression: AddResponseCompression() for Gzip/Brotli
  • Async/Await: Non-blocking I/O operations throughout the pipeline
  • EF Core: .AsNoTracking(), .Select() projections, compiled queries
  • HTTP/2 & HTTP/3: Multiplexing and reduced overhead (Kestrel native support)
  • Minimal APIs: Lower memory footprint than controller-based APIs

Practical Example

In one of my projects, we had a product catalog API that was returning full entities with navigation properties, causing 2-3 second response times. Here's how I optimized it:

// Before - Slow (loads all related data)
[HttpGet]
public async Task<IActionResult> GetProducts()
{
    var products = await _context.Products
        .Include(p => p.Category)
        .Include(p => p.Reviews)
        .ToListAsync();
    return Ok(products);
}

// After - Optimized
[HttpGet]
[ResponseCache(Duration = 300, VaryByQueryKeys = new[] { "page", "size" })]
public async Task<IActionResult> GetProducts(
    [FromQuery] int page = 1, 
    [FromQuery] int size = 20)
{
    var products = await _context.Products
        .AsNoTracking() // No change tracking overhead
        .Select(p => new ProductDto // Project only needed fields
        {
            Id = p.Id,
            Name = p.Name,
            Price = p.Price,
            CategoryName = p.Category.Name
        })
        .Skip((page - 1) * size)
        .Take(size)
        .ToListAsync();
        
    return Ok(products);
}

// Program.cs - Enable compression
builder.Services.AddResponseCompression(options =>
{
    options.EnableForHttps = true;
    options.Providers.Add<BrotliCompressionProvider>();
    options.Providers.Add<GzipCompressionProvider>();
});

app.UseResponseCompression();
app.UseResponseCaching();

Result: Response time dropped from 2.5s to 180ms, and payload size reduced by 65%. We also implemented Redis distributed caching for frequently accessed endpoints.

Key Takeaway: Show you understand the full optimization stack—query optimization, caching strategies, compression, and async patterns. Mention real metrics (latency improvements, throughput gains) to demonstrate production experience.

38. Explain Versioning in API

Theory

API versioning allows you to evolve your API without breaking existing clients. It enables you to introduce new features, change response structures, or deprecate endpoints while maintaining backward compatibility. Common strategies include URL versioning, header versioning, query string versioning, and media type versioning.

How it works in .NET Core

The Asp.Versioning.Http NuGet package (formerly Microsoft.AspNetCore.Mvc.Versioning) provides comprehensive versioning support:

  • URL Path: /api/v1/products vs /api/v2/products
  • Query String: /api/products?api-version=1.0
  • Header: Custom header like X-API-Version: 1.0
  • Media Type: Accept: application/vnd.myapi.v1+json
  • ApiVersion Attribute: Decorates controllers/actions with [ApiVersion("1.0")]
  • Deprecation Support: Mark versions as deprecated with sunset dates

Practical Example

In one of my projects migrating from a monolith to microservices, we needed to version our APIs because mobile clients couldn't update immediately. Here's the setup I implemented:

// Program.cs - Configure versioning
builder.Services.AddApiVersioning(options =>
{
    options.DefaultApiVersion = new ApiVersion(1, 0);
    options.AssumeDefaultVersionWhenUnspecified = true;
    options.ReportApiVersions = true; // Adds api-supported-versions header
    options.ApiVersionReader = ApiVersionReader.Combine(
        new UrlSegmentApiVersionReader(),
        new HeaderApiVersionReader("X-API-Version"),
        new QueryStringApiVersionReader("api-version")
    );
}).AddApiExplorer(options =>
{
    options.GroupNameFormat = "'v'VVV"; // For Swagger docs
    options.SubstituteApiVersionInUrl = true;
});

// V1 Controller - Original implementation
[ApiController]
[Route("api/v{version:apiVersion}/[controller]")]
[ApiVersion("1.0")]
public class OrdersController : ControllerBase
{
    [HttpGet("{id}")]
    public async Task<ActionResult<OrderV1Dto>> GetOrder(int id)
    {
        // Old structure: flat order with items as array
        return new OrderV1Dto
        {
            OrderId = id,
            CustomerName = "John Doe",
            Items = new[] { "Item1", "Item2" }
        };
    }
}

// V2 Controller - Enhanced with better structure
[ApiController]
[Route("api/v{version:apiVersion}/[controller]")]
[ApiVersion("2.0")]
public class OrdersController : ControllerBase
{
    [HttpGet("{id}")]
    public async Task<ActionResult<OrderV2Dto>> GetOrder(int id)
    {
        // New structure: nested items with details + hypermedia links
        return new OrderV2Dto
        {
            OrderId = id,
            Customer = new CustomerDto { Name = "John Doe", Email = "john@example.com" },
            Items = new List<OrderItemDto>
            {
                new() { ProductId = 1, Name = "Item1", Quantity = 2, Price = 29.99m }
            },
            Links = new { Self = $"/api/v2/orders/{id}", Cancel = $"/api/v2/orders/{id}/cancel" }
        };
    }
}

// Mark V1 as deprecated
[ApiVersion("1.0", Deprecated = true)]

Result: Legacy clients continued working on v1 while new features rolled out on v2. We added Sunset headers to v1 responses warning of eventual deprecation.

Key Takeaway: Demonstrate you know multiple versioning strategies and can choose the right one (URL for public APIs, header for internal). Mention handling deprecation gracefully with clear communication to consumers.

39. Difference between POST and PUT

Theory

POST is used to create new resources and is non-idempotent (multiple identical requests create multiple resources). PUT is used to update or replace an existing resource and is idempotent (multiple identical requests produce the same result). POST typically returns 201 Created with location header, while PUT returns 200 OK or 204 No Content.

How it works in .NET Core

In ASP.NET Core Web API:

  • POST: Decorated with [HttpPost], usually no ID in route. Returns CreatedAtAction() or CreatedAtRoute()
  • PUT: Decorated with [HttpPut("{id}")], requires resource ID. Returns NoContent() or Ok()
  • Model Binding: Both use [FromBody] for request payload
  • Validation: Both leverage ModelState.IsValid and Data Annotations
  • Idempotency Keys: Can implement custom headers for POST idempotency using middleware

Practical Example

In one of my projects building an e-commerce API, we had clear separation between creating products (POST) and updating inventory (PUT):

// POST - Create a new product (non-idempotent)
[HttpPost]
[ProducesResponseType(StatusCodes.Status201Created)]
[ProducesResponseType(StatusCodes.Status400BadRequest)]
public async Task<IActionResult> CreateProduct([FromBody] CreateProductDto dto)
{
    if (!ModelState.IsValid)
        return BadRequest(ModelState);
    
    var product = new Product
    {
        Name = dto.Name,
        Price = dto.Price,
        CreatedAt = DateTime.UtcNow
    };
    
    _context.Products.Add(product);
    await _context.SaveChangesAsync();
    
    // Multiple calls = multiple products created
    return CreatedAtAction(
        nameof(GetProduct), 
        new { id = product.Id }, 
        product
    );
}

// PUT - Update existing product (idempotent)
[HttpPut("{id}")]
[ProducesResponseType(StatusCodes.Status204NoContent)]
[ProducesResponseType(StatusCodes.Status400BadRequest)]
[ProducesResponseType(StatusCodes.Status404NotFound)]
public async Task<IActionResult> UpdateProduct(int id, [FromBody] UpdateProductDto dto)
{
    if (!ModelState.IsValid)
        return BadRequest(ModelState);
    
    var product = await _context.Products.FindAsync(id);
    if (product == null)
        return NotFound();
    
    // Complete replacement of resource
    product.Name = dto.Name;
    product.Price = dto.Price;
    product.UpdatedAt = DateTime.UtcNow;
    
    await _context.SaveChangesAsync();
    
    // Multiple identical calls = same result
    return NoContent();
}

// Alternative: PUT for upsert (create or replace)
[HttpPut("{id}")]
public async Task<IActionResult> UpsertProduct(int id, [FromBody] ProductDto dto)
{
    var product = await _context.Products.FindAsync(id);
    var isNewResource = product == null;
    
    if (isNewResource)
    {
        product = new Product { Id = id };
        _context.Products.Add(product);
    }
    
    product.Name = dto.Name;
    product.Price = dto.Price;
    
    await _context.SaveChangesAsync();
    
    return isNewResource 
        ? CreatedAtAction(nameof(GetProduct), new { id }, product)
        : NoContent();
}

Real Issue I Faced: We had payment processing where clients would retry on network timeouts. POST requests would create duplicate orders. I solved this by implementing an idempotency key middleware that checked a Redis cache before processing POST requests.

Key Takeaway: Emphasize idempotency as the key difference—PUT can be safely retried, POST cannot. Mention real-world scenarios like payment processing or network retries where this matters critically.

40. Example HTTP PATCH request

Theory

HTTP PATCH is used for partial updates to a resource—modifying specific fields without replacing the entire entity. Unlike PUT (which replaces the whole resource), PATCH only sends changed fields. Common formats include JSON Patch (RFC 6902) and JSON Merge Patch (RFC 7396). PATCH is idempotent in most implementations.

How it works in .NET Core

.NET Core supports PATCH through:

  • [HttpPatch] attribute: Routes HTTP PATCH requests to specific actions
  • Microsoft.AspNetCore.JsonPatch: NuGet package implementing RFC 6902
  • JsonPatchDocument<T>: Type-safe patch document with operations (add, remove, replace, copy, move, test)
  • ApplyTo() method: Applies patch operations to target object with validation
  • ModelState integration: Validates patch operations before applying

Practical Example

In one of my projects with a user profile API, we had mobile clients that only wanted to update specific fields (like email or phone) without sending the entire profile. Here's the implementation:

// Install NuGet: Microsoft.AspNetCore.JsonPatch
// Install NuGet: Microsoft.AspNetCore.Mvc.NewtonsoftJson

// Program.cs - Enable Newtonsoft for PATCH support
builder.Services.AddControllers()
    .AddNewtonsoftJson();

// Controller
[ApiController]
[Route("api/[controller]")]
public class UsersController : ControllerBase
{
    private readonly AppDbContext _context;
    
    public UsersController(AppDbContext context) => _context = context;
    
    [HttpPatch("{id}")]
    [ProducesResponseType(StatusCodes.Status204NoContent)]
    [ProducesResponseType(StatusCodes.Status400BadRequest)]
    [ProducesResponseType(StatusCodes.Status404NotFound)]
    public async Task<IActionResult> PatchUser(
        int id, 
        [FromBody] JsonPatchDocument<UserUpdateDto> patchDoc)
    {
        if (patchDoc == null)
            return BadRequest("Patch document is null");
        
        var user = await _context.Users.FindAsync(id);
        if (user == null)
            return NotFound();
        
        // Map entity to DTO
        var userDto = new UserUpdateDto
        {
            Email = user.Email,
            PhoneNumber = user.PhoneNumber,
            FirstName = user.FirstName,
            LastName = user.LastName
        };
        
        // Apply patch with validation
        patchDoc.ApplyTo(userDto, ModelState);
        
        if (!ModelState.IsValid)
            return BadRequest(ModelState);
        
        // Validate the patched DTO
        if (!TryValidateModel(userDto))
            return BadRequest(ModelState);
        
        // Map DTO back to entity
        user.Email = userDto.Email;
        user.PhoneNumber = userDto.PhoneNumber;
        user.FirstName = userDto.FirstName;
        user.LastName = userDto.LastName;
        user.UpdatedAt = DateTime.UtcNow;
        
        await _context.SaveChangesAsync();
        
        return NoContent();
    }
}

public class UserUpdateDto
{
    [Required]
    [EmailAddress]
    public string Email { get; set; }
    
    [Phone]
    public string PhoneNumber { get; set; }
    
    [MaxLength(50)]
    public string FirstName { get; set; }
    
    [MaxLength(50)]
    public string LastName { get; set; }
}

Example Client Request (JSON Patch format):

PATCH /api/users/123 HTTP/1.1
Content-Type: application/json-patch+json

[
  {
    "op": "replace",
    "path": "/email",
    "value": "newemail@example.com"
  },
  {
    "op": "replace",
    "path": "/phoneNumber",
    "value": "+1-555-1234"
  }
]

Alternative: Simple PATCH (without JsonPatchDocument):

// Simpler approach for basic partial updates
[HttpPatch("{id}")]
public async Task<IActionResult> SimplePartialUpdate(
    int id, 
    [FromBody] Dictionary<string, object> updates)
{
    var user = await _context.Users.FindAsync(id);
    if (user == null) return NotFound();
    
    foreach (var kvp in updates)
    {
        var prop = typeof(User).GetProperty(kvp.Key);
        if (prop != null && prop.CanWrite)
        {
            var value = Convert.ChangeType(kvp.Value, prop.PropertyType);
            prop.SetValue(user, value);
        }
    }
    
    await _context.SaveChangesAsync();
    return NoContent();
}

// Client sends: { "email": "new@example.com", "phoneNumber": "555-1234" }

Why I Used This: Mobile bandwidth was expensive for users in emerging markets, so sending only changed fields reduced payload size by 80% compared to PUT. We also used ETags with If-Match headers to prevent concurrent update conflicts.

Key Takeaway: Show you understand JSON Patch RFC standard and when PATCH is superior to PUT (bandwidth constraints, partial updates, mobile clients). Mention validation at both patch document and model levels.
