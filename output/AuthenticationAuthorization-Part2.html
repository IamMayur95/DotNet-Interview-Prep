Q54: Different ways to authenticate and authorize users for a Web API
1. Theory
Web API authentication/authorization can be implemented using multiple approaches: JWT Bearer Tokens (stateless, scalable, best for SPAs and mobile apps), OAuth 2.0/OpenID Connect (delegated authorization via third-party providers like Azure AD, Google, Auth0), API Keys (simple but less secure, for service-to-service communication), Cookie-based Authentication (session-based, for traditional web apps), Windows Authentication (for intranet apps using Active Directory), Certificate-based Authentication (mutual TLS for high-security scenarios), and Basic Authentication (username/password encoded in headers, only over HTTPS). Modern APIs typically use JWT for internal services and OAuth 2.0 for external integrations. The choice depends on security requirements, client types (browser, mobile, server), and whether you need single sign-on (SSO) or federated identity.
2. How it Works in .NET Core
.NET Core supports multiple authentication schemes via middleware:
  • JWT Bearer - AddJwtBearer() for token-based auth
  • OAuth 2.0/OpenID Connect - AddOpenIdConnect(), AddGoogle(), AddMicrosoftAccount()
  • API Keys - Custom middleware or IAuthenticationHandler
  • Cookie Authentication - AddCookie() for session-based auth
  • Windows Authentication - AddNegotiate() for AD integration
  • Certificate Authentication - AddCertificate() for mutual TLS
  • Multiple Schemes - AddAuthentication().AddJwtBearer().AddCookie() with default scheme
3. Practical Example
In a B2B SaaS platform, we implemented multiple authentication methods: JWT tokens for the React SPA and mobile apps, OAuth 2.0 (Azure AD) for enterprise SSO customers, and API keys for server-to-server integrations. We used AddAuthentication() with multiple schemes and [Authorize(AuthenticationSchemes = ...)] to specify which endpoints accept which authentication methods. 
// Program.cs - Multiple Authentication Schemes

// Custom API Key Authentication Handler
using Microsoft.AspNetCore.Authentication;
using Microsoft.Extensions.Options;
using System.Security.Claims;
using System.Text.Encodings.Web;

public class ApiKeyAuthenticationOptions : AuthenticationSchemeOptions
{
    public string HeaderName { get; set; } = "X-API-Key";
}

public class ApiKeyAuthenticationHandler : AuthenticationHandler<ApiKeyAuthenticationOptions>
{
    private readonly IApiKeyValidator _apiKeyValidator;

    public ApiKeyAuthenticationHandler(
        IOptionsMonitor<ApiKeyAuthenticationOptions> options,
        ILoggerFactory logger,
        UrlEncoder encoder,
        IApiKeyValidator apiKeyValidator)
        : base(options, logger, encoder)
    {
        _apiKeyValidator = apiKeyValidator;
    }

    protected override async Task<AuthenticateResult> HandleAuthenticateAsync()
    {
        if (!Request.Headers.TryGetValue(Options.HeaderName, out var apiKeyHeaderValues))
        {
            return AuthenticateResult.NoResult(); // No API key provided
        }

        var apiKey = apiKeyHeaderValues.FirstOrDefault();
        if (string.IsNullOrEmpty(apiKey))
        {
            return AuthenticateResult.Fail("Invalid API Key");
        }

        var apiKeyInfo = await _apiKeyValidator.ValidateAsync(apiKey);
        if (apiKeyInfo == null)
        {
            return AuthenticateResult.Fail("Invalid API Key");
        }

        // Create claims principal
        var claims = new[]
        {
            new Claim(ClaimTypes.Name, apiKeyInfo.ClientName),
            new Claim(ClaimTypes.NameIdentifier, apiKeyInfo.ClientId),
            new Claim("api_key_id", apiKeyInfo.KeyId)
        };

        var identity = new ClaimsIdentity(claims, Scheme.Name);
        var principal = new ClaimsPrincipal(identity);
        var ticket = new AuthenticationTicket(principal, Scheme.Name);

        return AuthenticateResult.Success(ticket);
    }
}

public interface IApiKeyValidator
{
    Task<ApiKeyInfo?> ValidateAsync(string apiKey);
}

public class ApiKeyValidator : IApiKeyValidator
{
    private readonly AppDbContext _context;

    public ApiKeyValidator(AppDbContext context)
    {
        _context = context;
    }

    public async Task<ApiKeyInfo?> ValidateAsync(string apiKey)
    {
        // Hash the API key and look up in database
        var hashedKey = HashApiKey(apiKey);
        
        var storedKey = await _context.ApiKeys
            .FirstOrDefaultAsync(k => k.HashedKey == hashedKey && k.IsActive);

        if (storedKey == null) return null;

        return new ApiKeyInfo
        {
            KeyId = storedKey.Id.ToString(),
            ClientId = storedKey.ClientId,
            ClientName = storedKey.ClientName
        };
    }

    private string HashApiKey(string apiKey)
    {
        using var sha256 = SHA256.Create();
        var bytes = sha256.ComputeHash(Encoding.UTF8.GetBytes(apiKey));
        return Convert.ToBase64String(bytes);
    }
}

4. Key Takeaway

Q55: Simple code example in C# on how to implement OAuth 2.0
1. Theory
OAuth 2.0 is an authorization framework that enables third-party applications to obtain limited access to a user's resources without exposing credentials. It uses four roles: Resource Owner (user), Client (your app), Authorization Server (issues tokens, e.g., Azure AD, Google), and Resource Server (your API). The most common flow is Authorization Code Flow with PKCE: (1) Client redirects user to authorization server, (2) User logs in and consents, (3) Authorization server redirects back with an authorization code, (4) Client exchanges code for access token, (5) Client uses access token to call APIs. OAuth 2.0 provides authorization (what you can access), while OpenID Connect (built on OAuth 2.0) adds authentication (who you are) via ID tokens. Never store client secrets in frontend code—use PKCE for SPAs and mobile apps.
2. How it Works in .NET Core
Implementing OAuth 2.0 in .NET involves:
  • As OAuth Client - AddOpenIdConnect() or AddGoogle(), AddMicrosoftAccount()
  • As OAuth Server - Use IdentityServer, Duende IdentityServer, or Azure AD
  • Microsoft.AspNetCore.Authentication.OpenIdConnect - for OpenID Connect
  • Microsoft.Identity.Web - for Azure AD/Microsoft Identity Platform
  • PKCE Support - automatically enabled in modern libraries
  • Token Storage - SaveTokens = true stores access/refresh tokens in auth properties
3. Practical Example
In a healthcare portal, we implemented OAuth 2.0 with Azure AD for hospital staff and Google OAuth for patients. Staff used their enterprise Azure AD credentials for SSO, while patients logged in with Google accounts. This eliminated password management and leveraged existing identity providers.

public class GraphApiService
{
    private readonly HttpClient _httpClient;
    private readonly IHttpContextAccessor _httpContextAccessor;

    public GraphApiService(HttpClient httpClient, IHttpContextAccessor httpContextAccessor)
    {
        _httpClient = httpClient;
        _httpContextAccessor = httpContextAccessor;
    }

    public async Task<string> GetUserProfileAsync()
    {
        // Get access token from authenticated user
        var accessToken = await _httpContextAccessor.HttpContext
            .GetTokenAsync("access_token");

        if (string.IsNullOrEmpty(accessToken))
            throw new UnauthorizedAccessException("No access token available");

        // Call Microsoft Graph API
        _httpClient.DefaultRequestHeaders.Authorization = 
            new AuthenticationHeaderValue("Bearer", accessToken);

        var response = await _httpClient.GetAsync("https://graph.microsoft.com/v1.0/me");
        response.EnsureSuccessStatusCode();

        return await response.Content.ReadAsStringAsync();
    }
}

// Register service
builder.Services.AddHttpContextAccessor();

// Use in endpoint

    var profile = await graphService.GetUserProfileAsync();
    return Results.Ok(profile);

Q56: Authorization - Role-based authorization
1. Theory
Role-based authorization (RBAC) restricts access based on a user's assigned roles (e.g., Admin, Manager, User). Roles are typically stored as claims in the authentication token (JWT or cookie) with claim type ClaimTypes.Role or "role". In .NET, you protect endpoints using [Authorize(Roles = "Admin")] to allow only users with the specified role(s). Multiple roles can be specified with comma separation (Roles = "Admin,Manager"), which acts as an OR condition. RBAC is simpler than policy-based authorization but less flexible—you can't easily implement complex rules like "Admin OR (Manager AND verified)". For enterprise apps, roles often come from Active Directory groups or external identity providers (Azure AD, Auth0). The key advantage: easy to understand and implement. The limitation: doesn't support fine-grained permissions without combining with claims or policies.
2. How it Works in .NET Core
Role-based authorization in .NET:
  • [Authorize(Roles = "Admin")] - requires specific role(s)
  • User.IsInRole("Admin") - manually check role in code
  • ClaimTypes.Role - standard claim type for roles
  • ASP.NET Identity - UserManager.AddToRoleAsync() for role management
  • JWT tokens - include roles in payload: new Claim(ClaimTypes.Role, "Admin")
  • Azure AD - roles from app registrations or directory roles
  • Policy-based alternative - RequireRole() in authorization policies
3. Practical Example
In an inventory management system, we implemented role-based authorization with four roles: Admin (full access), Manager (read/write inventory), Staff (read inventory, create orders), and Viewer (read-only). Admins could manage users, Managers could approve orders, Staff could process orders, and Viewers could only view reports. We stored roles in JWT claims and enforced them using [Authorize(Roles)].

// JWT Token Generation with Roles
public class JwtService
{
    private readonly IConfiguration _configuration;

    public JwtService(IConfiguration configuration)
    {
        _configuration = configuration;
    }

    public string GenerateToken(User user, IEnumerable<string> roles)
    {
        var securityKey = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(_configuration["Jwt:SecretKey"]));
        
        var credentials = new SigningCredentials(securityKey, SecurityAlgorithms.HmacSha256);

        // Create role claims (one claim per role)
        var claims = new List<Claim>
        {
            new Claim(JwtRegisteredClaimNames.Sub, user.Id.ToString()),
            new Claim(JwtRegisteredClaimNames.Email, user.Email),
            new Claim(ClaimTypes.Name, user.Username)
        };

        // Add each role as a separate claim
        foreach (var role in roles)
        {
            claims.Add(new Claim(ClaimTypes.Role, role));
        }

        var token = new JwtSecurityToken(
            issuer: _configuration["Jwt:Issuer"],
            audience: _configuration["Jwt:Audience"],
            claims: claims,
            expires: DateTime.UtcNow.AddMinutes(15),
            signingCredentials: credentials
        );

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}

// Role Management Service
public class RoleManagementService
{
    private readonly UserManager<ApplicationUser> _userManager;
    private readonly RoleManager<IdentityRole> _roleManager;

    public RoleManagementService(
        UserManager<ApplicationUser> userManager,
        RoleManager<IdentityRole> roleManager)
    {
        _userManager = userManager;
        _roleManager = roleManager;
    }

    public async Task SeedRolesAsync()
    {
        var roles = new[] { "Admin", "Manager", "Staff", "Viewer" };

        foreach (var role in roles)
        {
            if (!await _roleManager.RoleExistsAsync(role))
            {
                await _roleManager.CreateAsync(new IdentityRole(role));
            }
        }
    }

    public async Task AssignRoleToUserAsync(string userId, string roleName)
    {
        var user = await _userManager.FindByIdAsync(userId);
        if (user == null) throw new Exception("User not found");

        if (!await _roleManager.RoleExistsAsync(roleName))
            throw new Exception("Role does not exist");

        await _userManager.AddToRoleAsync(user, roleName);
    }

    public async Task<IList<string>> GetUserRolesAsync(string userId)
    {
        var user = await _userManager.FindByIdAsync(userId);
        if (user == null) return new List<string>();

        return await _userManager.GetRolesAsync(user);
    }
}

// Seed roles on startup
var app = builder.Build();

{
    var roleService = scope.ServiceProvider.GetRequiredService<RoleManagementService>();
    await roleService.SeedRolesAsync();
}

4. Key Takeaway
Role-based authorization uses [Authorize(Roles = "Admin")] to restrict endpoint access. Roles are stored as ClaimTypes.Role claims in JWT or cookies. Use ASP.NET Identity's UserManager for role management. Multiple roles in [Authorize] act as OR (Admin OR Manager). For complex rules, combine with policy-based authorization. Always test role restrictions with automated tests.
