Database CI/CD in Entity Framework

📖 Theory

Database CI/CD with Entity Framework involves automating database schema changes through the deployment pipeline. Instead of manually running migrations in production, you script and version them, integrate them into your build/release process, and apply them automatically (or with approval gates). This ensures schema changes are tracked in source control, tested in lower environments, and deployed consistently. Key concerns include idempotency (safe to rerun), rollback strategies, and zero-downtime deployments.

⚙️ How It Works in .NET Core

EF Core migrations are the foundation of database CI/CD:

  • Migration Scripts: Use dotnet ef migrations add to generate migration files tracked in Git
  • SQL Script Generation: Use dotnet ef migrations script to create idempotent SQL scripts for deployment
  • Runtime Migrations: Call context.Database.Migrate() on app startup (simple but risky in production)
  • CI/CD Integration: Run migrations as part of Azure DevOps, GitHub Actions, or Jenkins pipelines
  • Tools:
    • EF Core Bundles (.NET 6+): Self-contained executables for migrations
    • DbUp or Flyway: Alternative migration tools with better control
    • Azure SQL Database Projects: State-based deployments with DACPAC files

Best practice: Generate SQL scripts in CI, review them, then apply via CD pipeline with proper credentials and approval gates.

💼 Practical Example

In a microservices project I worked on, we had 12 services each with their own database. Manual migrations were a nightmare. Here's how I automated it with GitHub Actions and EF Core: 

# .github/workflows/deploy.yml
name: Deploy API with Database Migration

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      # Build and generate migration SQL script
      - name: Generate Migration Script
        run: |
          dotnet tool install --global dotnet-ef
          dotnet ef migrations script --idempotent --output ./migration.sql \
            --project ./MyApi/MyApi.csproj
      
      # Review artifact (optional approval step)
      - name: Upload Migration Script
        uses: actions/upload-artifact@v3
        with:
          name: migration-script
          path: ./migration.sql
      
      # Apply migration to Azure SQL
      - name: Run Migration
        run: |
          sqlcmd -S ${{ secrets.SQL_SERVER }} \
            -U ${{ secrets.SQL_USER }} \
            -P ${{ secrets.SQL_PASSWORD }} \
            -d MyDatabase -i ./migration.sql

For .NET 6+, I also used EF Core Bundles for better portability: 

// Generate a self-contained migration bundle
dotnet ef migrations bundle --output ./efbundle --self-contained

// In CI/CD pipeline, just run the bundle
./efbundle --connection "Server=...;Database=MyDb;..."

This eliminated manual errors, gave us audit trails, and enabled rollback by running specific migration scripts in reverse.

🎯 Key Takeaway

Never run migrations manually in production. Use dotnet ef migrations script to generate idempotent SQL, integrate it into your CI/CD pipeline, and apply with approval gates. EF Core Bundles (.NET 6+) simplify this further. Show you understand trade-offs between runtime migrations (fast but risky) vs. scripted migrations (safer, auditable).

How We Can Deploy .NET Core App on Linux

📖 Theory

.NET Core is cross-platform, so deploying to Linux is straightforward. You publish the app as a self-contained or framework-dependent deployment, configure a process manager (like systemd) to keep it running, and place a reverse proxy (Nginx or Apache) in front of Kestrel for production. Linux hosting is popular because it's cost-effective, lightweight, and integrates well with Docker and cloud-native architectures.

⚙️ How It Works in .NET Core

Deployment steps for Linux:

  • Publish the App:
    • dotnet publish -c Release -o ./publish
    • Choose deployment mode: Framework-dependent (requires .NET runtime on server) or Self-contained (bundles runtime)
  • Copy to Server: Use scp, rsync, or CI/CD pipelines
  • Install .NET Runtime: If framework-dependent, install .NET SDK/runtime via apt, yum, or snap
  • Configure systemd: Create a service file to manage the app lifecycle
  • Set Up Reverse Proxy: Configure Nginx or Apache to forward requests to Kestrel (default port 5000/5001)
  • Security: Use non-root user, configure firewall (ufw/iptables), and set up HTTPS with Let's Encrypt

Kestrel is production-ready but Nginx/Apache add SSL termination, static file caching, and load balancing.

💼 Practical Example

I deployed a high-traffic ASP.NET Core API to Ubuntu servers behind Nginx. Here's the exact process I used: 

# 1. Publish the app locally
dotnet publish -c Release -r linux-x64 --self-contained false -o ./publish

# 2. Copy to server (via SCP or CI/CD)
scp -r ./publish/* user@server:/var/www/myapi

# 3. Create systemd service: /etc/systemd/system/myapi.service
[Unit]
Description=My ASP.NET Core API
After=network.target

[Service]
WorkingDirectory=/var/www/myapi
ExecStart=/usr/bin/dotnet /var/www/myapi/MyApi.dll
Restart=always
RestartSec=10
User=www-data
Environment=ASPNETCORE_ENVIRONMENT=Production
Environment=DOTNET_PRINT_TELEMETRY_MESSAGE=false

[Install]
WantedBy=multi-user.target

# 4. Enable and start the service
sudo systemctl enable myapi.service
sudo systemctl start myapi.service
sudo systemctl status myapi.service

# 5. Configure Nginx: /etc/nginx/sites-available/myapi
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://localhost:5000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection keep-alive;
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

# 6. Enable site and reload Nginx
sudo ln -s /etc/nginx/sites-available/myapi /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

# 7. Set up HTTPS with Let's Encrypt
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d api.example.com

This setup gave us auto-restart on crashes, proper logging via journalctl, and HTTPS at zero cost. For monitoring, I added Prometheus exporters and integrated with Azure Monitor.

🎯 Key Takeaway

Deploying to Linux is production-standard for .NET Core. Know the full stack: systemd for process management, Nginx for reverse proxy, and Let's Encrypt for SSL. Self-contained deployments simplify versioning; framework-dependent saves space. Show you've done this end-to-end, not just read docs.

How to Deploy ASP.NET App in Docker

📖 Theory

Docker containerizes your ASP.NET app with all its dependencies into a portable image that runs consistently across environments. You create a Dockerfile that defines the build and runtime steps, build the image, and deploy it to Docker Hub, Azure Container Registry, or AWS ECR. Containers solve "works on my machine" problems and are foundational for Kubernetes, microservices, and CI/CD pipelines.

⚙️ How It Works in .NET Core

Docker deployment steps:

  • Dockerfile: Use official Microsoft images (mcr.microsoft.com/dotnet/aspnet for runtime, dotnet/sdk for builds)
  • Multi-Stage Builds: Build in SDK image, copy artifacts to runtime image (reduces size ~10x)
  • Build Image: docker build -t myapi:latest .
  • Run Locally: docker run -p 8080:80 myapi:latest
  • Push to Registry: docker push myregistry.azurecr.io/myapi:latest
  • Deploy: Use Docker Compose, Azure Container Instances, App Service, or Kubernetes
  • Best Practices:
    • Use .dockerignore to exclude bin/obj folders
    • Run as non-root user for security
    • Use health checks and environment variables for config

💼 Practical Example

In a microservices migration, I containerized 20+ ASP.NET Core APIs. Here's the production-grade Dockerfile I used: 

# Multi-stage Dockerfile
# Stage 1: Build
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# Copy csproj and restore (cached layer)
COPY ["MyApi/MyApi.csproj", "MyApi/"]
RUN dotnet restore "MyApi/MyApi.csproj"

# Copy source and build
COPY . .
WORKDIR "/src/MyApi"
RUN dotnet build "MyApi.csproj" -c Release -o /app/build

# Stage 2: Publish
FROM build AS publish
RUN dotnet publish "MyApi.csproj" -c Release -o /app/publish /p:UseAppHost=false

# Stage 3: Runtime
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS final
WORKDIR /app
EXPOSE 80
EXPOSE 443

# Security: Run as non-root
RUN adduser --disabled-password --gecos '' appuser && chown -R appuser /app
USER appuser

COPY --from=publish /app/publish .

# Health check
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD curl --fail http://localhost/health || exit 1

ENTRYPOINT ["dotnet", "MyApi.dll"]

Docker Compose for local dev: 

# docker-compose.yml
version: '3.8'
services:
  api:
    build: .
    ports:
      - "8080:80"
    environment:
      - ASPNETCORE_ENVIRONMENT=Development
      - ConnectionStrings__DefaultConnection=Server=db;Database=MyDb;...
    depends_on:
      - db
  
  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      - ACCEPT_EULA=Y
      - SA_PASSWORD=YourStrong@Passw0rd
    ports:
      - "1433:1433"

CI/CD with GitHub Actions: 

# Push to Azure Container Registry and deploy
- name: Build and Push Docker Image
  run: |
    docker build -t myregistry.azurecr.io/myapi:${{ github.sha }} .
    docker push myregistry.azurecr.io/myapi:${{ github.sha }}
    
- name: Deploy to Azure Container Apps
  run: |
    az containerapp update \
      --name myapi --resource-group myRG \
      --image myregistry.azurecr.io/myapi:${{ github.sha }}

This cut deployment time from 20 minutes to under 2 minutes and eliminated environment drift entirely.

🎯 Key Takeaway

Multi-stage Dockerfiles are non-negotiable—they keep images lean (runtime only) and builds fast (cached layers). Always use official Microsoft base images and run as non-root. Show you understand the full workflow: Dockerfile → Build → Push → Deploy to AKS/Azure Container Apps/AWS ECS.

Explain Docker and Kubernetes

📖 Theory

Docker is a containerization platform that packages apps with their dependencies into isolated containers. It solves the "works on my machine" problem by ensuring consistency across dev, test, and production. Kubernetes (K8s) is a container orchestration platform that manages Docker containers at scale—handling deployment, scaling, load balancing, self-healing, and rolling updates. While Docker runs containers, Kubernetes runs clusters of containers across multiple machines.

⚙️ How It Works in .NET Core

Docker:

  • Containerizes .NET apps using Dockerfile
  • Provides lightweight, portable runtime environments
  • Works with Docker Desktop (local), Docker Compose (multi-container), and registries (ACR, Docker Hub)

Kubernetes:

  • Pods: Smallest unit, runs one or more containers
  • Deployments: Manage replica sets, rolling updates, and rollbacks
  • Services: Expose pods via stable IPs and DNS (ClusterIP, LoadBalancer, NodePort)
  • Ingress: HTTP/HTTPS routing and SSL termination
  • ConfigMaps/Secrets: Inject configuration and credentials
  • Helm: Package manager for K8s (like NuGet for deployments)
  • Managed Services: Azure Kubernetes Service (AKS), AWS EKS, Google GKE

For .NET developers, Kubernetes means deploying containerized APIs/services to a cluster that auto-scales, self-heals, and handles traffic distribution.

💼 Practical Example

In a fintech project, we migrated 30 microservices from VMs to AKS (Azure Kubernetes Service). Here's how I deployed a .NET Core API: 

# Step 1: Build and push Docker image (see previous answer)
docker build -t myregistry.azurecr.io/payment-api:v1.0 .
docker push myregistry.azurecr.io/payment-api:v1.0

# Step 2: Kubernetes Deployment YAML (deployment.yaml)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-api
spec:
  replicas: 3  # Run 3 instances for high availability
  selector:
    matchLabels:
      app: payment-api
  template:
    metadata:
      labels:
        app: payment-api
    spec:
      containers:
      - name: payment-api
        image: myregistry.azurecr.io/payment-api:v1.0
        ports:
        - containerPort: 80
        env:
        - name: ASPNETCORE_ENVIRONMENT
          value: "Production"
        - name: ConnectionStrings__DefaultConnection
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: connectionString
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 80
          initialDelaySeconds: 30
          periodSeconds: 10

# Step 3: Service YAML (service.yaml)
apiVersion: v1
kind: Service
metadata:
  name: payment-api-service
spec:
  type: LoadBalancer
  selector:
    app: payment-api
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80

# Step 4: Apply to AKS cluster
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

# Step 5: Check status
kubectl get pods
kubectl get services
kubectl logs -l app=payment-api

# Step 6: Scale dynamically
kubectl scale deployment payment-api --replicas=10

# Step 7: Rolling update (zero downtime)
kubectl set image deployment/payment-api payment-api=myregistry.azurecr.io/payment-api:v1.1
kubectl rollout status deployment/payment-api

Using Helm for reusable deployments: 

# Create Helm chart
helm create payment-api-chart

# Deploy with custom values
helm install payment-api ./payment-api-chart \
  --set image.tag=v1.0 \
  --set replicaCount=5

# Upgrade
helm upgrade payment-api ./payment-api-chart --set image.tag=v1.1

Kubernetes handled auto-scaling during peak loads (Black Friday), auto-restarted crashed pods, and enabled zero-downtime deployments. We went from 20-minute manual deployments to 2-minute automated rollouts.

🎯 Key Takeaway

Docker = container runtime; Kubernetes = container orchestration at scale. For .NET Core, you containerize with Docker and deploy/manage with Kubernetes. Know core K8s concepts: Pods, Deployments, Services, Ingress, ConfigMaps/Secrets. Show you've worked with AKS, EKS, or on-prem K8s—not just theory.
