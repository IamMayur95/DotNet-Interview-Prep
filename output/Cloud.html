Q1: What are the types of Azure Functions?

📚 Theory

Azure Functions are serverless compute services that run event-driven code without managing infrastructure. There are different types based on hosting plans (Consumption, Premium, App Service) and trigger types (HTTP, Timer, Queue, Blob, Event Grid, Service Bus, Cosmos DB, etc.). Additionally, Durable Functions enable stateful workflows in a serverless environment.

⚙️ How It Works in .NET Core

Hosting Plans:

  • Consumption Plan: Pay-per-execution, auto-scales, 5-minute timeout default
  • Premium Plan: Pre-warmed instances, VNET integration, unlimited execution time
  • App Service Plan: Dedicated VMs, always-on, predictable pricing

Function Types by Trigger:

  • HTTP Triggered: HttpTrigger attribute, REST API endpoints
  • Timer Triggered: TimerTrigger with CRON expressions
  • Queue/Service Bus: QueueTrigger, ServiceBusTrigger
  • Blob/Event Grid: BlobTrigger, EventGridTrigger
  • Durable Functions: Orchestrator, Activity, Entity functions using Microsoft.Azure.WebJobs.Extensions.DurableTask

💼 Practical Example

In one of my projects, we migrated a batch processing system to Azure Functions. We used a Timer Triggered function on a Premium plan to process nightly reports, an HTTP Triggered function for real-time API requests, and Durable Functions for a multi-step approval workflow that required state management across days. 

// HTTP Triggered Function
[FunctionName("GetUserData")]
public static async Task<IActionResult> Run(
    [HttpTrigger(AuthorizationLevel.Function, "get", "post")] HttpRequest req,
    ILogger log)
{
    log.LogInformation("HTTP trigger function processed a request.");
    return new OkObjectResult("User data fetched");
}

// Timer Triggered Function (runs daily at 2 AM)
[FunctionName("DailyReportProcessor")]
public static void Run(
    [TimerTrigger("0 0 2 * * *")] TimerInfo myTimer,
    ILogger log)
{
    log.LogInformation($"Timer trigger function executed at: {DateTime.Now}");
    // Process daily reports
}

🎯 Key Takeaway

Azure Functions come in three hosting plans (Consumption, Premium, App Service) and support multiple trigger types (HTTP, Timer, Queue, Blob, etc.). For stateful workflows, use Durable Functions. Choose the plan based on performance, cost, and scaling requirements.

Q2: If an Azure Key Vault secret is used in a connection string and that secret reaches its expiration date, what happens to the application using it?

📚 Theory

When an Azure Key Vault secret expires, it becomes disabled and cannot be retrieved by applications. Any application attempting to access the expired secret will receive an error (typically HTTP 403 Forbidden), causing connection failures. The application won't automatically refresh the secret unless you implement secret rotation or periodic reload mechanisms. This can lead to downtime if not handled proactively.

⚙️ How It Works in .NET Core

Key Vault Integration:

  • Use Azure.Extensions.AspNetCore.Configuration.Secrets NuGet package
  • Configure in Program.cs using AddAzureKeyVault()
  • Secrets are loaded at startup by default (no automatic refresh)

Handling Expiration:

  • ConfigureReload: Enable periodic reload with ConfigureKeyVault(options => options.ReloadInterval = TimeSpan.FromMinutes(30))
  • Secret Rotation: Use Azure Key Vault's built-in rotation with Event Grid notifications
  • Monitoring: Set up Azure Monitor alerts before expiration dates

💼 Practical Example

In one of my projects, we had a production outage because a SQL connection string secret expired overnight. After that incident, I implemented automatic secret refresh every 30 minutes and configured Azure Monitor alerts 30 days before expiration. We also used Managed Identity to avoid storing credentials. 

// Program.cs - Configure Key Vault with auto-refresh
var builder = WebApplication.CreateBuilder(args);

var keyVaultUrl = new Uri(builder.Configuration["KeyVaultUrl"]);
var credential = new DefaultAzureCredential();

builder.Configuration.AddAzureKeyVault(keyVaultUrl, credential, new AzureKeyVaultConfigurationOptions
{
    // Reload secrets every 30 minutes
    ReloadInterval = TimeSpan.FromMinutes(30)
});

var app = builder.Build();

// Access secret
var connectionString = app.Configuration["ConnectionStrings:SqlDatabase"];
// If expired, this will be null or throw exception

🎯 Key Takeaway

Expired Key Vault secrets cause application failures (403 errors). Implement automatic secret rotation, configure periodic reload in .NET Core using ReloadInterval, and set up expiration alerts to prevent downtime. Always test secret rotation in non-production environments first.

Q3: Explain the process of whitelisting an IP using Bastion and how connection from VM was established? Explain the VNET creation.

📚 Theory

Azure Bastion provides secure RDP/SSH access to VMs without exposing them to the public internet. It acts as a jump server within a VNET. IP whitelisting is done via Network Security Groups (NSGs) on the Bastion subnet or VM subnet. A VNET (Virtual Network) is an isolated network in Azure where you define subnets, IP ranges, and security rules. Bastion requires a dedicated AzureBastionSubnet with a minimum /27 CIDR block.

⚙️ How It Works in .NET Core

VNET Creation Steps:

  • Create a VNET with address space (e.g., 10.0.0.0/16)
  • Add subnets: AzureBastionSubnet (e.g., 10.0.1.0/27) and VM subnet (e.g., 10.0.2.0/24)
  • Deploy Azure Bastion in the AzureBastionSubnet

IP Whitelisting via NSG:

  • Create NSG rule: Allow inbound traffic from specific IPs on port 443 (Bastion uses HTTPS)
  • Attach NSG to AzureBastionSubnet or VM subnet
  • Add outbound rules to allow Bastion to communicate with VMs (port 3389 for RDP, 22 for SSH)

Connection Process:

  • User connects to Azure Portal → Navigate to VM → Click "Connect" → Select "Bastion"
  • Bastion validates NSG rules, establishes SSL connection over port 443
  • No public IP needed on VM; connection is proxied through Bastion

💼 Practical Example

In one of my projects, we deployed a .NET Core API on Azure VMs and needed secure access for DevOps teams. We created a VNET with two subnets: one for Bastion (10.10.1.0/27) and one for VMs (10.10.2.0/24). We whitelisted our office IP range (203.0.113.0/24) in the NSG attached to the Bastion subnet. This allowed only our team to connect via Bastion to the VMs for troubleshooting without exposing RDP ports publicly. 

// Azure CLI commands to set up VNET and Bastion

# 1. Create Resource Group
az group create --name MyResourceGroup --location eastus

# 2. Create VNET with address space
az network vnet create \
  --name MyVNet \
  --resource-group MyResourceGroup \
  --address-prefix 10.10.0.0/16 \
  --subnet-name VMSubnet \
  --subnet-prefix 10.10.2.0/24

# 3. Create AzureBastionSubnet
az network vnet subnet create \
  --name AzureBastionSubnet \
  --resource-group MyResourceGroup \
  --vnet-name MyVNet \
  --address-prefix 10.10.1.0/27

# 4. Create Public IP for Bastion
az network public-ip create \
  --resource-group MyResourceGroup \
  --name BastionPublicIP \
  --sku Standard

# 5. Create Bastion
az network bastion create \
  --name MyBastion \
  --resource-group MyResourceGroup \
  --vnet-name MyVNet \
  --public-ip-address BastionPublicIP

# 6. Create NSG and whitelist IP
az network nsg create --resource-group MyResourceGroup --name BastionNSG
az network nsg rule create \
  --resource-group MyResourceGroup \
  --nsg-name BastionNSG \
  --name AllowOfficeIP \
  --priority 100 \
  --source-address-prefixes 203.0.113.0/24 \
  --destination-port-ranges 443 \
  --access Allow \
  --protocol Tcp

🎯 Key Takeaway

Azure Bastion provides secure VM access without public IPs. Create a VNET with AzureBastionSubnet (/27 minimum), deploy Bastion, and whitelist IPs using NSG rules on port 443. Connection is proxied through Bastion over SSL, eliminating the need for public RDP/SSH ports.

Q4: How will you trigger ADF pipeline?

📚 Theory

Azure Data Factory (ADF) pipelines can be triggered in multiple ways: Manual execution, Schedule triggers (CRON-based), Tumbling Window triggers (fixed intervals with dependency), Event-based triggers (Blob storage events), and REST API/SDK for programmatic execution. Each method suits different scenarios—scheduled batch jobs, real-time data ingestion, or orchestration from external systems.

⚙️ How It Works in .NET Core

Trigger Types:

  • Manual: Click "Trigger Now" in Azure Portal or use Azure CLI
  • Schedule Trigger: Define CRON expression in ADF (e.g., daily at 2 AM)
  • Tumbling Window: Fixed-size, non-overlapping time windows with retry logic
  • Event Trigger: Responds to Blob storage events (file created/deleted)
  • REST API: Call ADF REST API from .NET Core using Azure.ResourceManager.DataFactory SDK

.NET Core Integration:

  • Use DataFactoryManagementClient from Microsoft.Azure.Management.DataFactory
  • Authenticate using DefaultAzureCredential or Service Principal
  • Call Pipelines.CreateRunAsync() to trigger programmatically

💼 Practical Example

In one of my projects, we had a .NET Core API that processed user uploads and needed to trigger an ADF pipeline to transform and load data into a data warehouse. We used event-based triggers for immediate processing when files were uploaded to Blob storage, and scheduled triggers for nightly aggregations. For ad-hoc runs, we triggered pipelines via REST API from our .NET Core admin portal. 

// Trigger ADF Pipeline from .NET Core using REST API
using Azure.Identity;
using Azure.ResourceManager;
using Azure.ResourceManager.DataFactory;

public class AdfTriggerService
{
    private readonly DataFactoryPipelineResource _pipeline;

    public AdfTriggerService(string subscriptionId, string resourceGroup, 
                              string factoryName, string pipelineName)
    {
        var credential = new DefaultAzureCredential();
        var armClient = new ArmClient(credential);
        
        var resourceId = DataFactoryPipelineResource.CreateResourceIdentifier(
            subscriptionId, resourceGroup, factoryName, pipelineName);
        
        _pipeline = armClient.GetDataFactoryPipelineResource(resourceId);
    }

    public async Task<string> TriggerPipelineAsync()
    {
        var runResponse = await _pipeline.CreateRunAsync();
        return runResponse.Value.RunId;
    }
}

// Usage in API Controller
[HttpPost("trigger-etl")]
public async Task<IActionResult> TriggerEtl()
{
    var service = new AdfTriggerService(
        "sub-123", "myResourceGroup", "myDataFactory", "EtlPipeline");
    
    var runId = await service.TriggerPipelineAsync();
    return Ok(new { RunId = runId });
}

🎯 Key Takeaway

ADF pipelines can be triggered manually, via schedules (CRON), tumbling windows (fixed intervals), event-based triggers (Blob events), or programmatically using REST API/.NET SDK. Use DataFactoryManagementClient in .NET Core to trigger pipelines dynamically from your applications.

Q5: Differentiate between Azure Topics and Queues

📚 Theory

Both Azure Queues and Topics are messaging services, but they differ in architecture. Queues follow a point-to-point pattern (one sender, one receiver), where each message is processed by a single consumer. Topics follow a publish-subscribe pattern, where messages are sent to multiple subscribers through subscriptions with optional filters. Topics enable fanout scenarios and decoupled multi-consumer architectures.

⚙️ How It Works in .NET Core

Azure Queue (Storage Queue or Service Bus Queue):

  • Use Case: Load leveling, task distribution, single consumer
  • Pattern: FIFO (First-In-First-Out), one message → one consumer
  • .NET Package: Azure.Messaging.ServiceBus or Azure.Storage.Queues
  • Message Guarantee: At-least-once delivery with peek-lock

Azure Topics (Service Bus Topics):

  • Use Case: Pub-sub, event broadcasting, multiple consumers
  • Pattern: One message → multiple subscriptions (each subscription gets a copy)
  • .NET Package: Azure.Messaging.ServiceBus
  • Filtering: SQL filters, correlation filters on subscriptions

💼 Practical Example

In one of my projects, we used Service Bus Queues for processing orders (one order → one processor), ensuring each order was handled exactly once. For the same system, we used Topics to broadcast "OrderPlaced" events to multiple services—inventory service, notification service, and analytics service—each with its own subscription. This allowed decoupled, parallel processing of the same event. 

// Sending to Queue (Point-to-Point)
var client = new ServiceBusClient(connectionString);
var sender = client.CreateSender("orderqueue");
await sender.SendMessageAsync(new ServiceBusMessage("Order123"));

// Publishing to Topic (Pub-Sub)
var topicSender = client.CreateSender("ordertopic");
await topicSender.SendMessageAsync(new ServiceBusMessage("Order123"));

// Multiple subscriptions receive the message:
// - "inventory-subscription" → updates stock
// - "notification-subscription" → sends email
// - "analytics-subscription" → logs metrics

🎯 Key Takeaway

Use Queues for point-to-point communication (one sender → one receiver) and Topics for publish-subscribe (one sender → multiple receivers). Topics support subscriptions with filters, enabling fanout patterns. Both use Azure.Messaging.ServiceBus in .NET Core.

Q6: Differentiate between Azure Logic Apps and Function Apps

📚 Theory

Azure Logic Apps is a low-code/no-code workflow automation service with 400+ built-in connectors for integrations (Salesforce, SAP, Office 365, etc.). It's designed for orchestration and integration scenarios. Azure Functions is a code-first serverless compute service for executing custom business logic in response to events. Logic Apps are billed per action execution, while Functions are billed by execution time and memory.

⚙️ How It Works in .NET Core

Azure Logic Apps:

  • Development: Visual designer (Azure Portal, VS Code), JSON workflow definition
  • Use Case: Integration with SaaS apps, B2B workflows, scheduled automation
  • Connectors: 400+ built-in (SQL, Blob, Office 365, Dynamics, ServiceNow)
  • Pricing: Per action execution (e.g., $0.000025 per action)
  • Limitations: Less control over code, vendor lock-in

Azure Functions:

  • Development: Code in C#, Python, Node.js, Java using VS/VS Code
  • Use Case: Custom business logic, data processing, API backends
  • Triggers: HTTP, Timer, Queue, Blob, Event Grid, Cosmos DB, etc.
  • Pricing: Consumption (per execution + memory), Premium, App Service
  • Advantages: Full control, testable, version control (Git)

💼 Practical Example

In one of my projects, we used Logic Apps to automate approval workflows—when a document was uploaded to SharePoint, it triggered a Logic App that sent approval emails via Outlook, updated Dynamics 365, and logged events in SQL. For the same project, we used Azure Functions to process uploaded PDFs (text extraction, validation) because it required custom C# code with third-party libraries (iTextSharp). Logic Apps for integration, Functions for custom logic. 

// Azure Function for Custom PDF Processing
[FunctionName("ProcessPdf")]
public static async Task Run(
    [BlobTrigger("uploads/{name}.pdf")] Stream pdfStream,
    string name,
    ILogger log)
{
    log.LogInformation($"Processing PDF: {name}");
    
    // Custom business logic
    var text = ExtractTextFromPdf(pdfStream);
    var isValid = ValidateContent(text);
    
    if (isValid)
    {
        // Store in database, trigger Logic App for approval
        await TriggerLogicApp(name, text);
    }
}

// Logic Apps Workflow (Visual Designer - JSON definition):
// 1. When PDF is validated (HTTP trigger from Function)
// 2. Send email via Office 365 Connector
// 3. Update Dynamics 365 record
// 4. Log to SQL Database

🎯 Key Takeaway

Use Logic Apps for low-code integration workflows with built-in connectors (Office 365, SAP, Salesforce). Use Function Apps for custom business logic requiring full code control. They complement each other—Logic Apps for orchestration, Functions for compute. Both are serverless and event-driven.
