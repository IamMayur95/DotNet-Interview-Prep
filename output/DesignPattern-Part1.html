Q85: What do I need to do to make API scalable? What design patterns should be followed?

📚 Theory

API scalability means handling increased load (more users, requests, data) without degrading performance. Key strategies include horizontal scaling (add more servers), stateless design (no session data on API servers), caching (reduce DB hits), async processing (offload long tasks), and database optimization. Design patterns like CQRS (separate read/write operations), Repository (abstract data access), and Circuit Breaker (prevent cascading failures) are essential for building scalable APIs.

⚙️ How it Works in .NET Core

Infrastructure Level:

  • Load Balancers: Azure Load Balancer, Application Gateway, or NGINX to distribute traffic
  • Caching: Redis (IDistributedCache), In-Memory Cache, Response Caching Middleware
  • Message Queues: Azure Service Bus, RabbitMQ for async processing
  • Database: Read replicas, connection pooling, indexes, query optimization

Code-Level Patterns:

  • Repository Pattern: Abstracts data access, enables testing and swapping data sources
  • CQRS: Separate read (queries) from write (commands) for optimization
  • Circuit Breaker: Polly library to handle transient failures gracefully
  • API Gateway: Ocelot or Azure API Management for routing, rate limiting, authentication
  • Asynchronous APIs: Use async/await everywhere, background jobs with Hangfire/Quartz

💼 Practical Example

In a high-traffic e-commerce API I worked on, we went from 100 req/sec to 10,000 req/sec. Here's what we implemented: 

// 1. STATELESS DESIGN - Store JWT in client, no server sessions
public class Startup
{
    public void ConfigureServices(IServiceCollection services)
    {
        // JWT-based auth (stateless)
        services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
            .AddJwtBearer(options => { /* config */ });
        
        // Redis distributed cache
        services.AddStackExchangeRedisCache(options =>
        {
            options.Configuration = "localhost:6379";
        });
    }
}

// 2. CACHING STRATEGY - Redis for expensive queries
public class ProductService
{
    private readonly IDistributedCache _cache;
    private readonly IProductRepository _repo;
    
    public async Task<Product> GetProductById(int id)
    {
        var cacheKey = $"product:{id}";
        var cached = await _cache.GetStringAsync(cacheKey);
        
        if (cached != null)
            return JsonSerializer.Deserialize<Product>(cached);
        
        var product = await _repo.GetByIdAsync(id);
        
        await _cache.SetStringAsync(cacheKey, 
            JsonSerializer.Serialize(product),
            new DistributedCacheEntryOptions { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10) });
        
        return product;
    }
}

// 3. ASYNC PROCESSING - Offload heavy tasks to queue
public class OrderController : ControllerBase
{
    private readonly IServiceBusClient _serviceBus;
    
    [HttpPost("orders")]
    public async Task<IActionResult> CreateOrder([FromBody] Order order)
    {
        // Save to DB quickly
        await _orderRepo.CreateAsync(order);
        
        // Send to queue for processing (email, inventory, etc.)
        await _serviceBus.SendMessageAsync("order-processing", order);
        
        return Accepted(new { orderId = order.Id }); // Return immediately
    }
}

// 4. CIRCUIT BREAKER PATTERN - Using Polly
public class PaymentService
{
    private readonly IAsyncPolicy<HttpResponseMessage> _circuitBreaker;
    
    public PaymentService()
    {
        _circuitBreaker = Policy
            .HandleResult<HttpResponseMessage>(r => !r.IsSuccessStatusCode)
            .CircuitBreakerAsync(3, TimeSpan.FromSeconds(30));
    }
    
    public async Task<bool> ProcessPayment(decimal amount)
    {
        var response = await _circuitBreaker.ExecuteAsync(async () =>
            await _httpClient.PostAsync("payment-gateway/charge", /* data */));
        
        return response.IsSuccessStatusCode;
    }
}

// 5. DATABASE OPTIMIZATION - Connection pooling, async, indexes
public class Startup
{

    {
        services.AddDbContext<AppDbContext>(options =>
            options.UseSqlServer(connectionString, sqlOptions =>

// 6. RESPONSE CACHING MIDDLEWARE

[ResponseCache(Duration = 300, Location = ResponseCacheLocation.Any)]
public async Task<IActionResult> GetProducts()

🎯 Key Takeaway

Scalability requires stateless design, caching (Redis), async processing (queues), load balancing, and database optimization. Use patterns like Repository, CQRS, Circuit Breaker, and API Gateway. Always measure with load testing (JMeter, k6) and monitor with Application Insights or Prometheus. Horizontal scaling works only if your API is stateless.

Q86: Explain repository pattern in .NET Core

📚 Theory

The Repository Pattern abstracts data access logic by creating an interface between the business logic and data layer. It provides a collection-like interface for accessing domain objects, hiding database-specific code (EF Core, Dapper, ADO.NET). Benefits include testability (mock repositories), separation of concerns, and easier data source swapping. Often paired with Unit of Work pattern to manage transactions across multiple repositories.

⚙️ How it Works in .NET Core

Typical structure:

  • IRepository<T>: Generic interface with CRUD methods (GetById, GetAll, Add, Update, Delete)
  • Repository<T>: Generic implementation using DbContext
  • IUnitOfWork: Manages DbContext lifetime and SaveChanges() across multiple repos
  • Specific Repositories: Like IProductRepository with custom queries (GetProductsInStock)

Register in Startup.cs with dependency injection: services.AddScoped<IProductRepository, ProductRepository>()

⚠️ Debate: Some argue EF Core's DbSet already implements repository pattern, so adding another layer is over-engineering. Use repositories when you need abstraction for testing or plan to swap data sources.

💼 Practical Example

In a multi-tenant SaaS application, I used the repository pattern to abstract EF Core, making it easy to write unit tests and later switch some data access to Dapper for performance-critical reads. 

// 1. Generic Repository Interface
public interface IRepository<T> where T : class
{
    Task<T> GetByIdAsync(int id);
    Task<IEnumerable<T>> GetAllAsync();
    Task<T> AddAsync(T entity);
    Task UpdateAsync(T entity);
    Task DeleteAsync(int id);
}

// 2. Generic Repository Implementation
public class Repository<T> : IRepository<T> where T : class
{
    protected readonly AppDbContext _context;
    protected readonly DbSet<T> _dbSet;
    
    public Repository(AppDbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }
    
    public virtual async Task<T> GetByIdAsync(int id)
    {
        return await _dbSet.FindAsync(id);
    }
    
    public virtual async Task<IEnumerable<T>> GetAllAsync()
    {
        return await _dbSet.ToListAsync();
    }
    
    public virtual async Task<T> AddAsync(T entity)
    {
        await _dbSet.AddAsync(entity);
        return entity;
    }
    
    public virtual async Task UpdateAsync(T entity)
    {
        _dbSet.Update(entity);
        await Task.CompletedTask;
    }
    
    public virtual async Task DeleteAsync(int id)
    {
        var entity = await GetByIdAsync(id);
        if (entity != null)
            _dbSet.Remove(entity);
    }
}

// 3. Specific Repository with Custom Methods
public interface IProductRepository : IRepository<Product>
{
    Task<IEnumerable<Product>> GetProductsInStockAsync();
    Task<Product> GetProductWithCategoryAsync(int id);
}

public class ProductRepository : Repository<Product>, IProductRepository
{
    public ProductRepository(AppDbContext context) : base(context) { }
    
    public async Task<IEnumerable<Product>> GetProductsInStockAsync()
    {
        return await _dbSet
            .Where(p => p.Stock > 0)
            .ToListAsync();
    }
    
    public async Task<Product> GetProductWithCategoryAsync(int id)
    {
        return await _dbSet
            .Include(p => p.Category)
            .FirstOrDefaultAsync(p => p.Id == id);
    }
}

// 4. Unit of Work Pattern (Manage transactions)
public interface IUnitOfWork : IDisposable
{
    IProductRepository Products { get; }
    IOrderRepository Orders { get; }
    Task<int> SaveChangesAsync();
}

public class UnitOfWork : IUnitOfWork
{
    private readonly AppDbContext _context;
    
    public IProductRepository Products { get; }
    public IOrderRepository Orders { get; }
    
    public UnitOfWork(AppDbContext context)
    {
        _context = context;
        Products = new ProductRepository(context);
        Orders = new OrderRepository(context);
    }
    
    public async Task<int> SaveChangesAsync()
    {
        return await _context.SaveChangesAsync();
    }
    
    public void Dispose()
    {
        _context.Dispose();
    }
}

// 5. Service Layer Using Repository
public class ProductService
{
    private readonly IUnitOfWork _unitOfWork;
    
    public ProductService(IUnitOfWork unitOfWork)
    {
        _unitOfWork = unitOfWork;
    }
    
    public async Task<Product> CreateProduct(Product product)
    {
        await _unitOfWork.Products.AddAsync(product);
        await _unitOfWork.SaveChangesAsync();
        return product;
    }
}

// 6. Dependency Injection Registration
public void ConfigureServices(IServiceCollection services)
{
    services.AddScoped<IUnitOfWork, UnitOfWork>();
    services.AddScoped<IProductRepository, ProductRepository>();
    services.AddScoped<ProductService>();
}

🎯 Key Takeaway

Repository pattern abstracts data access, making code testable and maintainable. Use generic repositories for common CRUD, specific repositories for custom queries, and Unit of Work for transaction management. Recognize that EF Core DbContext is already a repository, so only add this layer when you need abstraction for testing or switching data sources.

Q87: SOLID principles in .NET with example

📚 Theory

SOLID is an acronym for five design principles that make software more maintainable, flexible, and testable:

  • S - Single Responsibility Principle: A class should have one reason to change
  • O - Open/Closed Principle: Open for extension, closed for modification
  • L - Liskov Substitution Principle: Subtypes must be substitutable for their base types
  • I - Interface Segregation Principle: Don't force clients to depend on unused interfaces
  • D - Dependency Inversion Principle: Depend on abstractions, not concretions

⚙️ How it Works in .NET Core

.NET Core's architecture embraces SOLID through:

  • Dependency Injection: Built-in DI container promotes DIP
  • Interfaces: ILogger, IConfiguration, IHostedService follow ISP
  • Middleware Pipeline: OCP - add middleware without modifying existing code
  • Generic Constraints: LSP - ensure type safety in generics

💼 Practical Example

I'll demonstrate each SOLID principle with real-world scenarios from projects I've worked on: 

// ========== S - SINGLE RESPONSIBILITY PRINCIPLE ==========
// BAD: Class doing too many things
public class UserService
{
    public void RegisterUser(User user)
    {
        // Validate user
        // Save to database
        // Send email
        // Log activity
    }
}

// GOOD: Separate responsibilities
public class UserService
{
    private readonly IUserRepository _repo;
    private readonly IEmailService _emailService;
    private readonly ILogger _logger;
    
    public async Task RegisterUserAsync(User user)
    {
        await _repo.AddAsync(user); // Data access responsibility
        await _emailService.SendWelcomeEmailAsync(user); // Email responsibility
        _logger.LogInformation($"User {user.Email} registered"); // Logging responsibility
    }
}

// ========== O - OPEN/CLOSED PRINCIPLE ==========
// BAD: Modifying class for new payment methods
public class PaymentProcessor
{
    public void ProcessPayment(string type, decimal amount)
    {
        if (type == "CreditCard")
            // Process credit card
        else if (type == "PayPal")
            // Process PayPal
        // Adding new method requires modifying this class!
    }
}

// GOOD: Open for extension via abstraction
public interface IPaymentMethod
{
    Task ProcessPaymentAsync(decimal amount);
}

public class CreditCardPayment : IPaymentMethod
{
    public async Task ProcessPaymentAsync(decimal amount)
    {
        // Credit card logic
    }
}

public class PayPalPayment : IPaymentMethod
{
    public async Task ProcessPaymentAsync(decimal amount)
    {
        // PayPal logic
    }
}

public class PaymentProcessor
{
    public async Task ProcessPayment(IPaymentMethod paymentMethod, decimal amount)
    {
        await paymentMethod.ProcessPaymentAsync(amount);
        // Add new payment methods without modifying this class!
    }
}

// ========== L - LISKOV SUBSTITUTION PRINCIPLE ==========
// BAD: Subclass violates base class contract
public class Bird
{
    public virtual void Fly() { /* fly logic */ }
}

public class Penguin : Bird
{
    public override void Fly()
    {
        throw new NotImplementedException("Penguins can't fly!");
        // Violates LSP - can't substitute Penguin for Bird
    }
}

// GOOD: Proper abstraction
public abstract class Bird
{
    public abstract void Move();
}

public class Sparrow : Bird
{
    public override void Move() => Fly();
    private void Fly() { /* fly logic */ }
}

public class Penguin : Bird
{
    public override void Move() => Swim();
    private void Swim() { /* swim logic */ }
}

// ========== I - INTERFACE SEGREGATION PRINCIPLE ==========
// BAD: Fat interface forcing unnecessary implementations
public interface IWorker
{
    void Work();
    void Eat();
    void Sleep();
}

public class Robot : IWorker
{
    public void Work() { /* work */ }
    public void Eat() { throw new NotImplementedException(); } // Robots don't eat!
    public void Sleep() { throw new NotImplementedException(); }
}

// GOOD: Segregated interfaces
public interface IWorkable
{
    void Work();
}

public interface IFeedable
{
    void Eat();
}

public class Human : IWorkable, IFeedable
{
    public void Work() { /* work */ }
    public void Eat() { /* eat */ }
}

public class Robot : IWorkable
{
    public void Work() { /* work */ }
    // No need to implement Eat()
}

// ========== D - DEPENDENCY INVERSION PRINCIPLE ==========
// BAD: High-level module depends on low-level module
public class OrderService
{
    private readonly SqlServerRepository _repo = new SqlServerRepository();
    // Tightly coupled to SqlServer - can't test or swap easily
    
    public void PlaceOrder(Order order)
    {
        _repo.Save(order);
    }
}

// GOOD: Depend on abstraction
public interface IOrderRepository
{
    Task SaveAsync(Order order);
}

public class OrderService
{
    private readonly IOrderRepository _repo;
    
    public OrderService(IOrderRepository repo) // Injected via DI
    {
        _repo = repo;
    }
    
    public async Task PlaceOrderAsync(Order order)
    {
        await _repo.SaveAsync(order);
        // Can inject SqlServerRepo, MongoRepo, or MockRepo for testing
    }
}

// Register in Startup.cs
services.AddScoped<IOrderRepository, SqlServerRepository>();

🎯 Key Takeaway

SOLID principles are fundamental to writing clean, maintainable .NET code. SRP keeps classes focused, OCP enables extension via interfaces, LSP ensures proper inheritance, ISP avoids fat interfaces, and DIP with dependency injection makes code testable and flexible. These principles work together to create robust architectures.

Q88: Singleton - Ensure single instance usage

📚 Theory

The Singleton pattern ensures a class has only one instance throughout the application lifetime and provides a global access point to it. Common use cases: configuration managers, logging, caching, database connection pools. Critical concerns: thread safety (multiple threads creating instances simultaneously), lazy initialization (create only when needed), and testability (singletons can make unit testing difficult).

⚙️ How it Works in .NET Core

Traditional Implementation Methods:

  • Eager Initialization: Create instance at class load time
  • Lazy<T>: Thread-safe lazy initialization (recommended)
  • Double-Check Locking: Optimize with lock only when needed (avoid, use Lazy instead)
  • Static Constructor: CLR guarantees thread safety

Modern .NET Core Approach: Use built-in Dependency Injection with services.AddSingleton<T>(). This is preferred over manual singleton implementation because it's testable, follows DIP, and the DI container manages lifecycle.

💼 Practical Example

I've implemented singletons in various ways across different projects. Here are the patterns I've used: 

// ========== METHOD 1: LAZY<T> (RECOMMENDED MANUAL APPROACH) ==========
public sealed class ConfigurationManager
{
    private static readonly Lazy<ConfigurationManager> _instance = 
        new Lazy<ConfigurationManager>(() => new ConfigurationManager());
    
    public static ConfigurationManager Instance => _instance.Value;
    
    private ConfigurationManager()
    {
        // Load configuration from file/environment
        LoadSettings();
    }
    
    public string ConnectionString { get; private set; }
    
    private void LoadSettings()
    {
        ConnectionString = "Server=...;Database=...";
    }
}

// Usage
var config = ConfigurationManager.Instance;
var connStr = config.ConnectionString;

// ========== METHOD 2: STATIC CONSTRUCTOR (THREAD-SAFE) ==========
public sealed class Logger
{
    private static readonly Logger _instance = new Logger();
    
    // Static constructor ensures thread safety
    static Logger() { }
    
    private Logger()
    {
        // Initialize logging
    }
    
    public static Logger Instance => _instance;
    
    public void Log(string message)
    {
        Console.WriteLine($"[{DateTime.Now}] {message}");
    }
}

// ========== METHOD 3: DOUBLE-CHECK LOCKING (AVOID - USE LAZY INSTEAD) ==========
public sealed class CacheManager
{
    private static CacheManager _instance;
    private static readonly object _lock = new object();
    
    public static CacheManager Instance
    {
        get
        {
            if (_instance == null) // First check (no lock)
            {
                lock (_lock)
                {
                    if (_instance == null) // Second check (with lock)
                    {
                        _instance = new CacheManager();
                    }
                }
            }
            return _instance;
        }
    }
    
    private CacheManager() { }
}

// ========== METHOD 4: DEPENDENCY INJECTION (MODERN .NET CORE WAY) ==========

// 1. Define service interface and implementation
public interface IApplicationSettings
{
    string ApiKey { get; }
    int MaxRetries { get; }
}

public class ApplicationSettings : IApplicationSettings
{
    public string ApiKey { get; set; }
    public int MaxRetries { get; set; }
    
    public ApplicationSettings()
    {
        // Load from configuration
        ApiKey = "your-api-key";
        MaxRetries = 3;
    }
}

// 2. Register as Singleton in Startup.cs
public class Startup
{
    public void ConfigureServices(IServiceCollection services)
    {
        // Singleton - Single instance for entire app lifetime
        services.AddSingleton<IApplicationSettings, ApplicationSettings>();
        
        // Alternative: Register instance directly
        services.AddSingleton<IApplicationSettings>(new ApplicationSettings());
        
        // Alternative: Factory method
        services.AddSingleton<IApplicationSettings>(sp =>
        {
            var config = sp.GetRequiredService<IConfiguration>();
            return new ApplicationSettings
            {
                ApiKey = config["ApiKey"],
                MaxRetries = int.Parse(config["MaxRetries"])
            };
        });
    }
}

{

    public OrderService(IApplicationSettings settings)
    {
        _settings = settings; // Same instance injected everywhere
    }
    
    public void ProcessOrder()
    {
        var apiKey = _settings.ApiKey;

    }
}

// ========== REAL-WORLD EXAMPLE: DATABASE CONNECTION POOL ==========
public sealed class DatabaseConnectionPool
{
    private static readonly Lazy<DatabaseConnectionPool> _instance =
        new Lazy<DatabaseConnectionPool>(() => new DatabaseConnectionPool());
    
    private readonly ConcurrentBag<SqlConnection> _connections;
    private const int MaxConnections = 100;
    
    public static DatabaseConnectionPool Instance => _instance.Value;
    
    private DatabaseConnectionPool()
    {
        _connections = new ConcurrentBag<SqlConnection>();
    }
    
    public SqlConnection GetConnection()
    {
        if (_connections.TryTake(out var connection))
            return connection;
        
        if (_connections.Count < MaxConnections)
            return new SqlConnection("connection-string");
        
        throw new InvalidOperationException("Connection pool exhausted");
    }
    
    public void ReturnConnection(SqlConnection connection)
    {
        _connections.Add(connection);
    }
}

// ========== TESTING CONSIDERATION ==========
// Problem: Hard to test code that uses static singletons
public class OrderProcessor
{
    public void Process()
    {
        Logger.Instance.Log("Processing..."); // Can't mock Logger!
    }
}

// Solution: Use DI with singleton lifetime
public class OrderProcessor
{
    private readonly ILogger _logger;
    
    public OrderProcessor(ILogger logger)
    {
        _logger = logger; // Can inject mock logger in tests!
    }
    
    public void Process()
    {
        _logger.Log("Processing...");
    }
}

🎯 Key Takeaway

For manual singleton implementation, use Lazy<T> for thread-safe lazy initialization. However, in modern .NET Core applications, prefer Dependency Injection with AddSingleton() because it's testable, follows SOLID principles, and the DI container manages the lifecycle. Manual singletons are harder to test and violate dependency inversion. Use singletons for stateless services like configuration, logging, or caching—avoid for stateful objects shared across requests.
