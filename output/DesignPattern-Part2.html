Factory Pattern - Encapsulate Object Creation

📖 Theory

The Factory Pattern is a creational design pattern that provides an interface for creating objects without specifying their exact class. Instead of using new directly, you delegate object creation to a factory method or class. This decouples client code from concrete implementations, making it easier to swap or extend object types. It's particularly useful when the creation logic is complex, involves configuration, or when you need to return different subtypes based on input parameters.

⚙️ How It Works in .NET Core

In modern .NET, factories are commonly implemented in a few ways:

  • Factory Method Pattern: A method that returns an interface or base class type
  • Abstract Factory: Using interfaces like IServiceProvider or custom factory interfaces registered in DI
  • Delegate Factories: Using Func<T> injected via DI to defer object creation
  • Named Factories: Leveraging IHttpClientFactory, ILoggerFactory, or custom implementations registered with AddTransient/AddScoped

.NET's built-in DI container supports factory delegates out of the box. You can register a Func<string, IService> that resolves different implementations based on runtime parameters.

💼 Practical Example

In one of my projects, we had multiple payment processors (Stripe, PayPal, Square) and needed to select the right one based on user preference. Instead of littering if/switch statements everywhere, I built a factory: 

// Interface
public interface IPaymentProcessor
{
    Task<bool> ProcessPaymentAsync(decimal amount);
}

// Implementations
public class StripeProcessor : IPaymentProcessor
{
    public async Task<bool> ProcessPaymentAsync(decimal amount)
        => await /* Stripe API call */;
}

// Factory
public class PaymentProcessorFactory
{
    private readonly IServiceProvider _serviceProvider;

    public PaymentProcessorFactory(IServiceProvider serviceProvider)
    {
        _serviceProvider = serviceProvider;
    }

    public IPaymentProcessor Create(string processorType)
    {
        return processorType switch
        {
            "Stripe" => _serviceProvider.GetRequiredService<StripeProcessor>(),
            "PayPal" => _serviceProvider.GetRequiredService<PayPalProcessor>(),
            _ => throw new ArgumentException("Invalid processor")
        };
    }
}

// DI Registration in Program.cs
builder.Services.AddTransient<StripeProcessor>();
builder.Services.AddTransient<PayPalProcessor>();
builder.Services.AddSingleton<PaymentProcessorFactory>();

This made unit testing trivial and allowed us to add new processors without touching existing code—just register them in DI and update the factory switch.

🎯 Key Takeaway

The Factory Pattern keeps your code loosely coupled and testable. In .NET Core, leverage the DI container to inject factories rather than hardcoding new everywhere. It's essential for scalable, maintainable systems where object creation logic needs flexibility.

CQRS - Command-Query Separation

📖 Theory

CQRS (Command Query Responsibility Segregation) is an architectural pattern that separates read operations (queries) from write operations (commands). Commands change state but don't return data; queries return data but don't change state. This separation allows you to optimize each side independently—scale reads differently from writes, use different data models, or even separate databases. It's particularly powerful in event-driven or domain-driven design (DDD) systems where complex business logic benefits from clear responsibility boundaries.

⚙️ How It Works in .NET Core

In .NET, CQRS is typically implemented using:

  • MediatR: A popular NuGet package (MediatR) that provides in-process messaging for commands and queries
  • Command/Query Handlers: Separate classes implementing IRequestHandler<TRequest, TResponse>
  • Separate Data Models: DTOs for reads (queries) vs domain entities for writes (commands)
  • Separate Databases (optional): Read replicas, Redis caches, or separate SQL/NoSQL stores for queries
  • Event Sourcing: Often paired with CQRS, using frameworks like EventStore or Azure Event Hubs

MediatR handles routing commands/queries to their handlers, making controllers thin and focused on HTTP concerns only.

💼 Practical Example

In a high-traffic e-commerce API I worked on, we had a bottleneck where product searches were slowing down order placements. I introduced CQRS using MediatR: 

// Command - Places an order (write)
public record PlaceOrderCommand(int UserId, List<int> ProductIds) 
    : IRequest<int>;

public class PlaceOrderHandler : IRequestHandler<PlaceOrderCommand, int>
{
    private readonly ApplicationDbContext _context;

    public async Task<int> Handle(PlaceOrderCommand request, CancellationToken ct)
    {
        var order = new Order { UserId = request.UserId, OrderDate = DateTime.UtcNow };
        _context.Orders.Add(order);
        await _context.SaveChangesAsync(ct);
        return order.Id;
    }
}

// Query - Gets order details (read)
public record GetOrderQuery(int OrderId) : IRequest<OrderDto>;

public class GetOrderHandler : IRequestHandler<GetOrderQuery, OrderDto>
{
    private readonly IReadOnlyDbContext _readDb; // Could be a Redis cache or read replica

    public async Task<OrderDto> Handle(GetOrderQuery request, CancellationToken ct)
    {
        return await _readDb.Orders
            .Where(o => o.Id == request.OrderId)
            .Select(o => new OrderDto(o.Id, o.Total))
            .FirstOrDefaultAsync(ct);
    }
}

// Controller
[ApiController]
[Route("api/orders")]
public class OrdersController : ControllerBase
{
    private readonly IMediator _mediator;

    [HttpPost]
    public async Task<IActionResult> PlaceOrder(PlaceOrderCommand command)
        => Ok(await _mediator.Send(command));

    [HttpGet("{id}")]
    public async Task<IActionResult> GetOrder(int id)
        => Ok(await _mediator.Send(new GetOrderQuery(id)));
}

We later moved queries to a Redis cache synced via an event bus. Performance improved 10x, and the codebase became much easier to reason about.

🎯 Key Takeaway

CQRS isn't overkill—it's essential for complex domains or high-scale systems. In .NET Core, MediatR makes implementation clean and testable. Show you understand when to use it (complex business logic, event sourcing, scale) vs. when it's over-engineering (simple CRUD apps).

Dependency Injection Pattern - Inversion of Control

📖 Theory

Dependency Injection (DI) is a design pattern that implements Inversion of Control (IoC), where dependencies are provided to a class rather than the class creating them itself. Instead of new Service() inside a class, you inject IService via constructor, properties, or methods. This decouples your code, making it testable, maintainable, and adhering to the SOLID principles—especially Dependency Inversion. A DI container manages object lifetimes and wires up dependencies automatically at runtime.

⚙️ How It Works in .NET Core

.NET Core has built-in DI through Microsoft.Extensions.DependencyInjection. Key concepts:

  • Service Lifetimes:
    • Transient: New instance every time it's requested
    • Scoped: One instance per HTTP request (or scope)
    • Singleton: One instance for the app's lifetime
  • Registration: Done in Program.cs (or Startup.cs in older versions) using builder.Services.Add*
  • Constructor Injection: The container automatically resolves dependencies declared in constructors
  • IServiceProvider: The underlying container that resolves services at runtime
  • Third-party Containers: You can replace the default with Autofac, Ninject, or others if needed

In minimal APIs and controllers, DI is automatic. In console apps or workers, you manually build an IHost or ServiceProvider.

💼 Practical Example

In a real-time notification system I built, we needed to send alerts via email, SMS, and push notifications. Instead of tight coupling, I used DI with the strategy pattern: 

// Service interface
public interface INotificationService
{
    Task SendAsync(string message);
}

// Implementations
public class EmailNotificationService : INotificationService
{
    public async Task SendAsync(string message)
        => await /* Send email */;
}

public class SmsNotificationService : INotificationService
{
    public async Task SendAsync(string message)
        => await /* Send SMS */;
}

// Consumer
public class AlertService
{
    private readonly IEnumerable<INotificationService> _notifiers;

    public AlertService(IEnumerable<INotificationService> notifiers)
    {
        _notifiers = notifiers;
    }

    public async Task BroadcastAlertAsync(string message)
    {
        foreach (var notifier in _notifiers)
        {
            await notifier.SendAsync(message);
        }
    }
}

// DI Registration in Program.cs
var builder = WebApplication.CreateBuilder(args);

// Register multiple implementations
builder.Services.AddTransient<INotificationService, EmailNotificationService>();
builder.Services.AddTransient<INotificationService, SmsNotificationService>();
builder.Services.AddScoped<AlertService>();

var app = builder.Build();

The container automatically injected all INotificationService implementations into AlertService. For unit tests, I could mock INotificationService easily without touching production code. When we added push notifications later, it was just one new registration—no changes to AlertService.

🎯 Key Takeaway

Dependency Injection is non-negotiable in modern .NET. Understand service lifetimes (Transient/Scoped/Singleton) cold—interviewers will ask when to use each. DI makes your code testable and loosely coupled, which is critical for maintainability and scaling teams.
