Q1: EF Core DbContext is scoped - why? What if I make it singleton? What will be the issues?

📚 Theory

DbContext is scoped by default because it's designed to represent a single unit of work with the database. It maintains change tracking, caches entities, and is not thread-safe. Making it singleton would cause multiple requests to share the same context, leading to concurrency issues, stale data, memory leaks (cached entities never released), and data corruption in multi-threaded scenarios.

⚙️ How It Works in .NET Core

Why Scoped?

  • Unit of Work Pattern: One DbContext per HTTP request, disposed after request completes
  • Change Tracker: Tracks entity states (Added, Modified, Deleted) per request
  • Connection Management: Opens/closes connections per request lifecycle
  • Thread Safety: Each request gets its own instance, avoiding race conditions

Problems with Singleton DbContext:

  • Concurrency Issues: Multiple threads accessing the same context simultaneously
  • Stale Data: First-level cache returns outdated entities across requests
  • Memory Leaks: Change tracker never clears, accumulating entities indefinitely
  • Connection Pooling Issues: Single connection shared across requests
  • SaveChanges Conflicts: Multiple requests trying to save changes on the same context

Registration:

// Correct - Scoped (default)
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(connectionString));

// WRONG - Singleton (will cause issues)
builder.Services.AddSingleton<AppDbContext>();

💼 Practical Example

In one of my projects, a junior developer accidentally registered DbContext as singleton to "improve performance." We immediately saw InvalidOperationException errors like "A second operation started on this context before a previous operation completed." Users were seeing each other's data due to shared change tracking. After reverting to scoped, all issues disappeared. The lesson: never make DbContext singleton—the connection pool already handles performance optimization. 

// Problem Scenario with Singleton DbContext
public class OrderService
{
    private readonly AppDbContext _context;

    public OrderService(AppDbContext context)
    {
        _context = context; // If singleton, shared across all requests!
    }

    public async Task CreateOrderAsync(Order order)
    {
        _context.Orders.Add(order);
        await _context.SaveChangesAsync(); 
        // Thread-safety violation if multiple requests call this simultaneously!
    }
}

// Correct Usage - Scoped DbContext
// Each HTTP request gets a fresh DbContext instance
// Disposed automatically at end of request
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(connectionString), 
    ServiceLifetime.Scoped); // Default, explicit for clarity

🎯 Key Takeaway

DbContext must be scoped (per-request) because it's not thread-safe and maintains change tracking state. Singleton DbContext causes concurrency issues, stale data, memory leaks, and data corruption. Always use AddDbContext() which registers it as scoped by default. Connection pooling handles performance optimization.

Q2: What are Scoped, Singleton, and Transient lifetimes in .NET Core? Explain the differences.

📚 Theory

.NET Core's DI container supports three service lifetimes: Transient (created each time requested), Scoped (one per request/scope), and Singleton (one instance for app lifetime). The lifetime determines when instances are created, shared, and disposed. Choosing the wrong lifetime can cause memory leaks, concurrency issues, or performance problems.

⚙️ How It Works in .NET Core

Lifetime Creation Scope Use Case Thread Safety
Transient Every injection No sharing Lightweight, stateless services Not needed
Scoped Once per request HTTP request DbContext, UnitOfWork Not needed
Singleton Once per app Application lifetime Configuration, caching, logging Required

Key Differences:

  • Transient: New instance every time → High memory usage if injected frequently
  • Scoped: Shared within request → Best for per-request state (DbContext)
  • Singleton: Shared globally → Must be thread-safe, no scoped dependencies

Registration:

// Transient - new instance every time
services.AddTransient<IEmailService, EmailService>();

// Scoped - once per HTTP request
services.AddScoped<IOrderService, OrderService>();

// Singleton - once per application lifetime
services.AddSingleton<ICacheService, CacheService>();

💼 Practical Example

In one of my projects, we had a NotificationService that sent emails. Initially registered as singleton, but it injected a scoped DbContext—this caused "Cannot consume scoped service from singleton" errors. We changed NotificationService to scoped. For a CacheService that stored app-wide data, we used singleton with thread-safe ConcurrentDictionary. Lightweight loggers were transient. 

// Example: Different Lifetimes

// 1. Transient - Created every time (lightweight)
public class GuidGenerator : IGuidGenerator
{
    public Guid Generate() => Guid.NewGuid();
}
services.AddTransient<IGuidGenerator, GuidGenerator>();

// 2. Scoped - Once per request (DbContext pattern)
public class OrderService : IOrderService
{
    private readonly AppDbContext _context;
    public OrderService(AppDbContext context) => _context = context;
}
services.AddScoped<IOrderService, OrderService>();

// 3. Singleton - Once per app (must be thread-safe)
public class CacheService : ICacheService
{
    private readonly ConcurrentDictionary<string, object> _cache = new();
    public void Set(string key, object value) => _cache[key] = value;
}
services.AddSingleton<ICacheService, CacheService>();

🎯 Key Takeaway

Transient = new instance every injection (lightweight services). Scoped = one per request (DbContext, UnitOfWork). Singleton = one per app lifetime (caching, config). Never inject scoped services into singletons. Singletons must be thread-safe. Choose based on state management needs.

Q3: What are different configuration providers in .NET Core?

📚 Theory

.NET Core uses a hierarchical configuration system that loads settings from multiple providers in order. Providers include appsettings.json, environment variables, command-line arguments, user secrets (development), Azure Key Vault, INI files, and XML files. Later providers override earlier ones, enabling environment-specific configuration (Dev, Staging, Prod).

⚙️ How It Works in .NET Core

Built-in Configuration Providers (in order):

  • appsettings.json: Base configuration file
  • appsettings.{Environment}.json: Environment-specific overrides (e.g., appsettings.Production.json)
  • User Secrets: Local development secrets (not in source control)
  • Environment Variables: OS-level settings (Docker, Azure App Service)
  • Command-Line Arguments: Runtime overrides (e.g., --ConnectionStrings:Default=...)
  • Azure Key Vault: Secure secrets management (via AddAzureKeyVault())
  • Custom Providers: Database, Redis, HTTP endpoints

Configuration Loading (default order):

// Program.cs - Default configuration setup
var builder = WebApplication.CreateBuilder(args);

// Automatically loads in this order:
// 1. appsettings.json
// 2. appsettings.{Environment}.json
// 3. User Secrets (if Development environment)
// 4. Environment Variables
// 5. Command-line arguments

// Add custom providers
builder.Configuration
    .AddJsonFile("custom-config.json", optional: true)
    .AddXmlFile("config.xml", optional: true)
    .AddAzureKeyVault(new Uri("https://myvault.vault.azure.net/"), 
                          new DefaultAzureCredential());

Accessing Configuration:

// Inject IConfiguration
public class MyService
{
    private readonly IConfiguration _config;
    
    public MyService(IConfiguration config) => _config = config;
    
    public void GetSettings()
    {
        var connString = _config["ConnectionStrings:Default"];
        var apiKey = _config.GetValue<string>("ApiKey");
        
        // Strongly-typed configuration
        var settings = _config.GetSection("AppSettings").Get<AppSettings>();
    }
}

💼 Practical Example

In one of my projects, we used appsettings.json for default settings, appsettings.Production.json for production overrides, Azure Key Vault for connection strings and API keys, and environment variables for Docker container configuration. During local development, we used User Secrets to avoid committing sensitive data. Command-line arguments were used for CI/CD pipelines to override specific settings during deployment. 

// appsettings.json (base)

  "Logging": { "LogLevel": { "Default": "Information" } },
  "ConnectionStrings": { "Default": "Server=localhost" }
}

// appsettings.Production.json (overrides)
{

}

// Environment Variable (Docker/Azure)

// User Secrets (Development - not in repo)
dotnet user-secrets set "ApiKey" "dev-secret-key-12345"

// Command-line override
dotnet run --ConnectionStrings:Default="Server=test-server"

🎯 Key Takeaway

.NET Core loads configuration from multiple providers in order: appsettings.json → environment-specific JSON → User Secrets → Environment Variables → Command-line. Later providers override earlier ones. Use Azure Key Vault for secrets in production. Access via IConfiguration or strongly-typed options pattern.

Q4: What are IHostedService and Hosted Services in .NET Core?

📚 Theory

IHostedService is an interface for running background tasks in .NET Core applications. It provides StartAsync() and StopAsync() methods that are called when the application starts and stops. BackgroundService is an abstract base class that simplifies implementing IHostedService with a ExecuteAsync() method for long-running operations. Use cases include scheduled jobs, message queue processors, and continuous data synchronization.

⚙️ How It Works in .NET Core

IHostedService Interface:

public interface IHostedService
{
    Task StartAsync(CancellationToken cancellationToken);
    Task StopAsync(CancellationToken cancellationToken);
}

BackgroundService (Recommended):

  • Abstract class that implements IHostedService
  • Override ExecuteAsync() for long-running work
  • Automatically handles start/stop lifecycle
  • Supports graceful shutdown via CancellationToken

Registration:

// Register hosted service
builder.Services.AddHostedService<DataSyncService>();

Use Cases:

  • Scheduled tasks (e.g., nightly report generation)
  • Message queue consumers (Azure Service Bus, RabbitMQ)
  • Data synchronization between systems
  • Cache warming on startup
  • Health check monitoring

💼 Practical Example

In one of my projects, we needed to process orders from an Azure Service Bus queue continuously. We implemented a BackgroundService that listened to the queue 24/7, processed messages, and updated the database. When the app restarted, StopAsync() ensured graceful shutdown (finishing current message before stopping). We also used IHostedService for a startup task that warmed up the cache with frequently accessed data. 

// BackgroundService Example - Queue Processor
public class OrderProcessorService : BackgroundService
{
    private readonly ILogger<OrderProcessorService> _logger;
    private readonly IServiceProvider _serviceProvider;

    public OrderProcessorService(
        ILogger<OrderProcessorService> logger, 
        IServiceProvider serviceProvider)
    {
        _logger = logger;
        _serviceProvider = serviceProvider;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _logger.LogInformation("Order Processor Service started");

        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                // Create scope for scoped services (DbContext)
                using var scope = _serviceProvider.CreateScope();
                var orderService = scope.ServiceProvider.GetRequiredService<IOrderService>();

                // Process orders from queue
                await orderService.ProcessPendingOrdersAsync();

                // Wait 10 seconds before next iteration
                await Task.Delay(TimeSpan.FromSeconds(10), stoppingToken);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Error processing orders");
                await Task.Delay(TimeSpan.FromSeconds(30), stoppingToken);
            }
        }

        _logger.LogInformation("Order Processor Service stopped");
    }
}

// Registration
builder.Services.AddHostedService<OrderProcessorService>();

🎯 Key Takeaway

IHostedService enables background tasks in .NET Core. Use BackgroundService (abstract base class) to implement long-running operations in ExecuteAsync(). Register with AddHostedService(). Use scopes for injecting scoped services like DbContext. Handle CancellationToken for graceful shutdown.

Q5: How do you make gRPC calls in .NET Core?

📚 Theory

gRPC is a high-performance RPC (Remote Procedure Call) framework using HTTP/2 and Protocol Buffers (Protobuf) for binary serialization. It's faster and more efficient than REST for microservices communication. .NET Core has built-in support via Grpc.AspNetCore (server) and Grpc.Net.Client (client). gRPC supports unary, server streaming, client streaming, and bidirectional streaming.

⚙️ How It Works in .NET Core

Steps to Implement gRPC:

  • 1. Define .proto file: Define service contract (methods, messages)
  • 2. Generate C# code: Auto-generated from .proto during build
  • 3. Implement service: Inherit from generated base class, override methods
  • 4. Configure server: Map gRPC services in Program.cs
  • 5. Create client: Use GrpcChannel and generated client stub

Required NuGet Packages:

  • Server: Grpc.AspNetCore
  • Client: Grpc.Net.Client, Google.Protobuf, Grpc.Tools

Advantages over REST:

  • 7-10x faster (binary serialization vs JSON)
  • Strongly-typed contracts
  • Bi-directional streaming
  • Built-in load balancing and deadlines

💼 Practical Example

In one of my projects, we had a microservices architecture where an Order Service needed to call an Inventory Service to check stock availability. We switched from REST to gRPC and saw a 60% reduction in latency. The strongly-typed contracts eliminated serialization errors at runtime. 

// 1. Define Proto File (inventory.proto)
syntax = "proto3";

option csharp_namespace = "InventoryService";

service Inventory {
  rpc CheckStock (StockRequest) returns (StockResponse);
}

message StockRequest {
  string product_id = 1;
  int32 quantity = 2;
}

message StockResponse {
  bool is_available = 1;
  int32 available_quantity = 2;
}

// 2. gRPC Service Implementation (Server)
public class InventoryService : Inventory.InventoryBase
{
    public override Task<StockResponse> CheckStock(
        StockRequest request, 
        ServerCallContext context)
    {
        // Business logic to check inventory
        var availableQty = GetStockFromDatabase(request.ProductId);
        
        return Task.FromResult(new StockResponse
        {
            IsAvailable = availableQty >= request.Quantity,
            AvailableQuantity = availableQty
        });
    }
}

// 3. Configure Server (Program.cs)
builder.Services.AddGrpc();
var app = builder.Build();
app.MapGrpcService<InventoryService>();

// 4. gRPC Client (Calling Service)
using var channel = GrpcChannel.ForAddress("https://localhost:5001");
var client = new Inventory.InventoryClient(channel);

var response = await client.CheckStockAsync(new StockRequest
{
    ProductId = "PROD-123",
    Quantity = 10
});

Console.WriteLine($"Available: {response.IsAvailable}, Qty: {response.AvailableQuantity}");

🎯 Key Takeaway

gRPC in .NET Core uses HTTP/2 and Protobuf for high-performance RPC. Define contracts in .proto files, implement services inheriting from generated base classes, configure with AddGrpc() and MapGrpcService(). Use GrpcChannel to create clients. 7-10x faster than REST with strongly-typed contracts.
