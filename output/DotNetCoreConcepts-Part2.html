Q6: How does ASP.NET Core handle Dependency Injection internally?

📚 Theory

ASP.NET Core has a built-in DI container (IServiceProvider) that manages service lifetimes, resolves dependencies, and handles object creation. It uses ServiceDescriptor to store service registrations (interface, implementation, lifetime). When a service is requested, the container checks the service collection, resolves dependencies recursively, creates instances based on lifetime (Transient/Scoped/Singleton), and injects them via constructor injection. Scoped services are managed per request using IServiceScopeFactory.

⚙️ How It Works in .NET Core

Internal Components:

  • IServiceCollection: Stores service descriptors (registrations) at startup
  • IServiceProvider: Runtime container that resolves and creates service instances
  • ServiceDescriptor: Defines service type, implementation type, and lifetime
  • IServiceScopeFactory: Creates scopes for scoped services (per HTTP request)
  • ActivatorUtilities: Creates instances and injects dependencies

Resolution Process:

  1. Service requested via constructor or GetService<T>()
  2. Container looks up ServiceDescriptor from IServiceCollection
  3. Checks lifetime: Singleton (return cached instance), Scoped (return from current scope), Transient (create new)
  4. Recursively resolves constructor dependencies
  5. Creates instance using ActivatorUtilities or factory delegate
  6. Tracks disposable services for cleanup (IDisposable)

Registration:

// Program.cs - Service Registration
var builder = WebApplication.CreateBuilder(args);

// Internally creates ServiceDescriptor and adds to IServiceCollection
builder.Services.AddTransient<IEmailService, EmailService>();
// Equivalent to:
// builder.Services.Add(new ServiceDescriptor(typeof(IEmailService), typeof(EmailService), ServiceLifetime.Transient));

builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddSingleton<ICacheService, CacheService>();

var app = builder.Build(); // Builds IServiceProvider from IServiceCollection

Resolution:

// Constructor Injection (recommended)
public class OrderController : ControllerBase
{
    private readonly IOrderService _orderService;
    private readonly IEmailService _emailService;

    // DI container automatically resolves and injects
    public OrderController(IOrderService orderService, IEmailService emailService)
    {
        _orderService = orderService;
        _emailService = emailService;
    }
}

// Manual Resolution (service locator pattern - avoid)
var emailService = app.Services.GetRequiredService<IEmailService>();

💼 Practical Example

In one of my projects, I debugged a "Cannot resolve scoped service from root provider" error. The issue was trying to resolve a scoped DbContext outside an HTTP request scope. I learned that ASP.NET Core creates a root IServiceProvider at startup (for singletons) and child scopes per request (for scoped services). Accessing scoped services from the root provider throws this error. Solution: use IServiceScopeFactory to create scopes manually in background services. 

// Problem: Accessing Scoped Service from Singleton
public class SingletonService
{
    private readonly AppDbContext _context; // WRONG! Scoped in Singleton
    
    public SingletonService(AppDbContext context) // Runtime error!
    {
        _context = context;
    }
}

// Solution: Use IServiceScopeFactory
public class SingletonService
{
    private readonly IServiceScopeFactory _scopeFactory;
    
    public SingletonService(IServiceScopeFactory scopeFactory)
    {
        _scopeFactory = scopeFactory;
    }
    
    public async Task DoWorkAsync()
    {
        using var scope = _scopeFactory.CreateScope();
        var context = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        // Use context safely
    }
}

🎯 Key Takeaway

ASP.NET Core's DI container uses IServiceCollection (registrations) and IServiceProvider (resolution). Services are stored as ServiceDescriptors with lifetimes. Container resolves dependencies recursively via constructor injection. Scoped services require IServiceScopeFactory. Never inject scoped services into singletons directly.

Q7: Explain Middleware in .NET Core and how to design custom middleware

📚 Theory

Middleware is software that sits between the HTTP request and response pipeline, processing requests before they reach controllers and responses before they return to clients. Each middleware component can short-circuit the pipeline (stop further processing) or pass control to the next middleware using next(). Middleware executes in the order it's registered. Examples: authentication, logging, exception handling, routing, CORS.

⚙️ How It Works in .NET Core

Middleware Pipeline:

Request → Middleware 1 → Middleware 2 → Middleware 3 → Controller/Endpoint
                 ↓              ↓              ↓              ↓
Response ← Middleware 1 ← Middleware 2 ← Middleware 3 ← Controller/Endpoint

Built-in Middleware (common order):

  1. UseExceptionHandler: Catches exceptions globally
  2. UseHttpsRedirection: Redirects HTTP to HTTPS
  3. UseStaticFiles: Serves static files (CSS, JS, images)
  4. UseRouting: Matches requests to endpoints
  5. UseAuthentication: Authenticates users
  6. UseAuthorization: Authorizes access
  7. UseEndpoints/MapControllers: Executes endpoint logic

Registering Middleware:

// Program.cs
var app = builder.Build();

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();

Custom Middleware (3 ways):

1. Inline with Use():

app.Use(async (context, next) =>
{
    // Before next middleware
    Console.WriteLine("Request received");
    
    await next(context); // Call next middleware
    
    // After next middleware
    Console.WriteLine("Response sent");
});

2. Extension Method (convention-based):

// Custom Middleware Class
public class RequestLoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger _logger;

    public RequestLoggingMiddleware(RequestDelegate next, ILogger<RequestLoggingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        _logger.LogInformation($"Request: {context.Request.Method} {context.Request.Path}");
        
        await _next(context); // Call next middleware
        
        _logger.LogInformation($"Response: {context.Response.StatusCode}");
    }
}

// Extension Method
public static class RequestLoggingMiddlewareExtensions
{
    public static IApplicationBuilder UseRequestLogging(this IApplicationBuilder app)
    {
        return app.UseMiddleware<RequestLoggingMiddleware>();
    }
}

// Usage
app.UseRequestLogging();

3. IMiddleware Interface (factory-based):

public class CorrelationIdMiddleware : IMiddleware
{
    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        context.Request.Headers.TryGetValue("X-Correlation-Id", out var correlationId);
        context.Response.Headers.Add("X-Correlation-Id", correlationId.ToString());
        
        await next(context);
    }
}

// Registration
builder.Services.AddTransient<CorrelationIdMiddleware>();
app.UseMiddleware<CorrelationIdMiddleware>();

💼 Practical Example

In one of my projects, we needed to log all API requests with execution time and add correlation IDs for distributed tracing. I created custom middleware that logged the request path, method, and response time. We also added middleware to validate API keys before authentication middleware. The order was critical: exception handling → request logging → API key validation → authentication → authorization → controllers. 

// Production-Ready Custom Middleware Example
public class PerformanceLoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<PerformanceLoggingMiddleware> _logger;

    public PerformanceLoggingMiddleware(RequestDelegate next, ILogger<PerformanceLoggingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        var stopwatch = Stopwatch.StartNew();
        
        try
        {
            await _next(context);
        }
        finally
        {
            stopwatch.Stop();
            _logger.LogInformation(
                "Request {Method} {Path} completed in {ElapsedMs}ms with status {StatusCode}",
                context.Request.Method,
                context.Request.Path,
                stopwatch.ElapsedMilliseconds,
                context.Response.StatusCode);
        }
    }
}

// Pipeline Configuration
app.UseExceptionHandler("/error"); // 1. Catch exceptions first
app.UseMiddleware<PerformanceLoggingMiddleware>(); // 2. Log performance
app.UseHttpsRedirection(); // 3. Force HTTPS
app.UseAuthentication(); // 4. Authenticate
app.UseAuthorization(); // 5. Authorize
app.MapControllers(); // 6. Execute endpoints

🎯 Key Takeaway

Middleware processes requests/responses in a pipeline. Create custom middleware with convention-based classes (RequestDelegate + InvokeAsync) or IMiddleware interface. Register with UseMiddleware<T>() or extension methods. Order matters: exception handling → logging → authentication → authorization → endpoints. Use next() to call the next middleware.

Q8: What is Configure and ConfigureServices (Startup.cs vs Program.cs)?

📚 Theory

In older .NET Core (2.x-5.x), Startup.cs contained two methods: ConfigureServices() for registering services in the DI container, and Configure() for building the HTTP request pipeline with middleware. In .NET 6+, this was simplified—Program.cs handles both using WebApplicationBuilder (services) and WebApplication (middleware). ConfigureServices → builder.Services, Configure → app.Use*().

⚙️ How It Works in .NET Core

Legacy Approach (Startup.cs in .NET Core 3.1-5.x):

// Startup.cs
public class Startup
{
    // 1. ConfigureServices - Register services (DI)
    public void ConfigureServices(IServiceCollection services)
    {
        services.AddControllers();
        services.AddDbContext<AppDbContext>(options =>
            options.UseSqlServer(Configuration.GetConnectionString("Default")));
        services.AddScoped<IOrderService, OrderService>();
    }

    // 2. Configure - Build middleware pipeline
    public void Configure(IApplicationBuilder app, IWebHostEnvironment env)
    {
        if (env.IsDevelopment())
        {
            app.UseDeveloperExceptionPage();
        }

        app.UseHttpsRedirection();
        app.UseRouting();
        app.UseAuthentication();
        app.UseAuthorization();
        app.UseEndpoints(endpoints =>
        {
            endpoints.MapControllers();
        });
    }
}

Modern Approach (Program.cs in .NET 6+):

// Program.cs (.NET 6+)
var builder = WebApplication.CreateBuilder(args);

// ConfigureServices equivalent - builder.Services
builder.Services.AddControllers();
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default")));
builder.Services.AddScoped<IOrderService, OrderService>();

var app = builder.Build(); // Creates IServiceProvider

// Configure equivalent - app.Use*()
if (app.Environment.IsDevelopment())
{
    app.UseDeveloperExceptionPage();
}

app.UseHttpsRedirection();
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();

app.Run();

Key Differences:

Aspect ConfigureServices (Old) builder.Services (New)
Purpose Register DI services Register DI services
Location Startup.cs Program.cs
Parameter IServiceCollection WebApplicationBuilder.Services
Aspect Configure (Old) app.Use* (New)
Purpose Build middleware pipeline Build middleware pipeline
Location Startup.cs Program.cs
Parameter IApplicationBuilder WebApplication

💼 Practical Example

When migrating a .NET Core 3.1 project to .NET 6, I moved all ConfigureServices code to builder.Services and all Configure code to app.Use*(). The biggest benefit was less boilerplate—no need for a separate Startup.cs class. Everything in Program.cs made the project structure simpler and easier to understand for new developers. 

// Real-World Program.cs Example (.NET 6+)
var builder = WebApplication.CreateBuilder(args);

// === ConfigureServices Section ===
// Add services to DI container
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

// Register custom services
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddSingleton<ICacheService, CacheService>();

// Add authentication
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options => { /* config */ });

var app = builder.Build();

// === Configure Section ===
// Build HTTP request pipeline (middleware)
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();
app.UseAuthentication(); // Must come before UseAuthorization
app.UseAuthorization();
app.MapControllers();

app.Run();

🎯 Key Takeaway

ConfigureServices registers DI services (old: Startup.cs, new: builder.Services). Configure builds middleware pipeline (old: Startup.cs, new: app.Use*()). .NET 6+ unified both in Program.cs for simplicity. ConfigureServices runs before Configure. Order matters in Configure for middleware.

Q9: What are Filters in .NET Core?

📚 Theory

Filters in .NET Core execute code at specific stages of the request processing pipeline, providing cross-cutting concerns like logging, authorization, caching, and exception handling. They run after routing but before/after action methods. There are 5 filter types: Authorization (runs first, checks permissions), Resource (caching, short-circuiting), Action (before/after action execution), Exception (handles exceptions), and Result (before/after result execution).

⚙️ How It Works in .NET Core

Filter Execution Order:

Request → Routing → Authorization Filter → Resource Filter → Action Filter 
   → Action Method → Result Filter → Exception Filter (if error) → Response

Filter Types:

Filter Type Interface Use Case Execution
Authorization IAuthorizationFilter Authorize requests First (before all)
Resource IResourceFilter Caching, short-circuit After authorization
Action IActionFilter Modify args/results Before/after action
Exception IExceptionFilter Handle exceptions On exception
Result IResultFilter Modify response Before/after result

Creating Filters:

// 1. Action Filter (most common)
public class LoggingActionFilter : IActionFilter
{
    private readonly ILogger _logger;

    public LoggingActionFilter(ILogger<LoggingActionFilter> logger)
    {
        _logger = logger;
    }

    // Executes before action method
    public void OnActionExecuting(ActionExecutingContext context)
    {
        _logger.LogInformation($"Executing action: {context.ActionDescriptor.DisplayName}");
    }

    // Executes after action method
    public void OnActionExecuted(ActionExecutedContext context)
    {
        _logger.LogInformation($"Executed action: {context.ActionDescriptor.DisplayName}");
    }
}

// 2. Exception Filter
public class CustomExceptionFilter : IExceptionFilter
{
    public void OnException(ExceptionContext context)
    {
        var error = new { message = context.Exception.Message };
        context.Result = new ObjectResult(error) { StatusCode = 500 };
        context.ExceptionHandled = true; // Prevent further exception propagation
    }
}

// 3. Authorization Filter
public class ApiKeyAuthFilter : IAuthorizationFilter
{
    public void OnAuthorization(AuthorizationFilterContext context)
    {
        if (!context.HttpContext.Request.Headers.TryGetValue("X-Api-Key", out var key))
        {
            context.Result = new UnauthorizedResult();
        }
    }
}

Registering Filters:

// Global (all controllers)
builder.Services.AddControllers(options =>
{
    options.Filters.Add<LoggingActionFilter>();
    options.Filters.Add<CustomExceptionFilter>();
});

// Controller-level (all actions in controller)
[ServiceFilter(typeof(LoggingActionFilter))]
public class OrderController : ControllerBase { }

// Action-level (specific action)
[ServiceFilter(typeof(ApiKeyAuthFilter))]
[HttpGet]
public IActionResult GetOrders() { }

💼 Practical Example

In one of my projects, we needed to validate API keys for specific endpoints, log execution time for all actions, and handle exceptions consistently. I created an Authorization Filter for API key validation, an Action Filter to measure execution time, and an Exception Filter to return standardized error responses. This eliminated repetitive code across 50+ controllers. 

// Production Example: Performance Logging Filter
public class PerformanceFilter : IActionFilter
{
    private readonly ILogger<PerformanceFilter> _logger;
    private Stopwatch _stopwatch;

    public PerformanceFilter(ILogger<PerformanceFilter> logger)
    {
        _logger = logger;
    }

    public void OnActionExecuting(ActionExecutingContext context)
    {
        _stopwatch = Stopwatch.StartNew();
    }

    public void OnActionExecuted(ActionExecutedContext context)
    {
        _stopwatch.Stop();
        _logger.LogInformation(
            "Action {Action} executed in {ElapsedMs}ms",
            context.ActionDescriptor.DisplayName,
            _stopwatch.ElapsedMilliseconds);
        
        if (_stopwatch.ElapsedMilliseconds > 1000)
        {
            _logger.LogWarning("Slow action detected!");
        }
    }
}

// Register globally
builder.Services.AddScoped<PerformanceFilter>();
builder.Services.AddControllers(options =>
{
    options.Filters.Add<PerformanceFilter>();
});

🎯 Key Takeaway

Filters execute at specific pipeline stages: Authorization → Resource → Action → Exception → Result. Use Action Filters for cross-cutting concerns (logging, validation), Exception Filters for error handling, Authorization Filters for security. Register globally (AddControllers), per-controller, or per-action using attributes.
