Q10: What is Localization and Globalization in .NET Core?

📚 Theory

Globalization (i18n) is designing apps to support multiple cultures and languages without code changes. Localization (l10n) is adapting apps for specific cultures by translating content, formatting dates/numbers, and adjusting for cultural conventions. .NET Core uses resource files (.resx) for translations, CultureInfo for culture-specific formatting, and RequestLocalizationMiddleware to detect user culture from query strings, cookies, or Accept-Language headers.

⚙️ How It Works in .NET Core

Key Concepts:

  • Globalization: Designing culture-neutral apps (e.g., using resource keys instead of hardcoded text)
  • Localization: Providing culture-specific resources (e.g., English, French, Spanish translations)
  • CultureInfo: Represents culture (e.g., en-US, fr-FR, es-ES) for formatting dates, numbers, currency
  • Resource Files: .resx files with key-value pairs for translations (e.g., Resources.en.resx, Resources.fr.resx)

Setup Steps:

  1. Install Package: Microsoft.Extensions.Localization
  2. Create Resource Files: SharedResource.en.resx, SharedResource.fr.resx
  3. Configure Services: AddLocalization() and AddRequestLocalization()
  4. Add Middleware: UseRequestLocalization()
  5. Inject IStringLocalizer: Use in controllers/views to retrieve localized strings

Configuration:

// Program.cs
var builder = WebApplication.CreateBuilder(args);

// 1. Add Localization Services
builder.Services.AddLocalization(options => options.ResourcesPath = "Resources");

// 2. Configure Supported Cultures
var supportedCultures = new[] { "en-US", "fr-FR", "es-ES" };
builder.Services.Configure<RequestLocalizationOptions>(options =>
{
    options.SetDefaultCulture(supportedCultures[0]);
    options.AddSupportedCultures(supportedCultures);
    options.AddSupportedUICultures(supportedCultures);
    
    // Culture detection order: Query String → Cookie → Accept-Language Header
    options.RequestCultureProviders.Insert(0, new QueryStringRequestCultureProvider());
});

builder.Services.AddControllers()
    .AddViewLocalization(); // For Razor views

var app = builder.Build();

// 3. Add Middleware (before routing)
app.UseRequestLocalization();
app.UseRouting();
app.MapControllers();

Resource Files (Resources/SharedResource.en.resx):

<!-- XML structure -->
<data name="Welcome" xml:space="preserve">
  <value>Welcome to our application</value>
</data>
<data name="Goodbye" xml:space="preserve">
  <value>Goodbye</value>
</data>

Resource Files (Resources/SharedResource.fr.resx):

<data name="Welcome" xml:space="preserve">
  <value>Bienvenue dans notre application</value>
</data>
<data name="Goodbye" xml:space="preserve">
  <value>Au revoir</value>
</data>

Usage in Controllers:

public class HomeController : ControllerBase
{
    private readonly IStringLocalizer<SharedResource> _localizer;

    public HomeController(IStringLocalizer<SharedResource> localizer)
    {
        _localizer = localizer;
    }

    [HttpGet]
    public IActionResult Index()
    {
        var welcomeMessage = _localizer["Welcome"]; // Auto-selects based on culture
        return Ok(new { message = welcomeMessage.ToString() });
    }
}

// Accessing: /api/home?culture=fr-FR
// Returns: { "message": "Bienvenue dans notre application" }

💼 Practical Example

In one of my projects, we built a multi-tenant SaaS app serving customers in US, France, and Spain. We used resource files for all UI text (validation messages, labels, emails). The culture was detected from user preferences stored in the database, passed via cookie. For dates and currency, we used CultureInfo to format correctly (e.g., $1,000.00 in US vs 1 000,00 € in France). This eliminated hardcoded strings and made adding new languages trivial. 

// Real-World Example: Date/Currency Formatting
public class OrderService
{
    private readonly IStringLocalizer<SharedResource> _localizer;

    public OrderService(IStringLocalizer<SharedResource> localizer)
    {
        _localizer = localizer;
    }

    public string GetOrderSummary(decimal total, DateTime orderDate)
    {
        var culture = CultureInfo.CurrentCulture;
        
        // Format currency based on culture
        var formattedTotal = total.ToString("C", culture);
        // en-US: $1,000.00
        // fr-FR: 1 000,00 €
        
        // Format date based on culture
        var formattedDate = orderDate.ToString("d", culture);
        // en-US: 12/31/2023
        // fr-FR: 31/12/2023
        
        // Get localized string
        var message = _localizer["OrderSummary", formattedTotal, formattedDate];
        return message;
    }
}

// Resource file (SharedResource.en.resx):
// Key: "OrderSummary"
// Value: "Your order total is {0} on {1}"

// Resource file (SharedResource.fr.resx):
// Key: "OrderSummary"
// Value: "Le total de votre commande est {0} le {1}"

🎯 Key Takeaway

Globalization = culture-neutral design. Localization = culture-specific content. Use .resx resource files for translations, IStringLocalizer for retrieving localized strings, and UseRequestLocalization() middleware to detect culture from query strings, cookies, or headers. CultureInfo handles date/number/currency formatting automatically.

Q11: What is the difference between Use() and Run() in .NET Core middleware?

📚 Theory

Use() adds middleware that can pass control to the next middleware by calling next(), allowing the pipeline to continue. Run() adds terminal middleware that short-circuits the pipeline—it never calls next(), so subsequent middleware is never executed. Use() for chaining middleware, Run() for final response handlers. There's also Map() for branching pipelines based on request paths.

⚙️ How It Works in .NET Core

Key Differences:

Method Calls next()? Purpose Use Case
Use() Yes (optional) Chain middleware Logging, auth, request processing
Run() No (terminal) End pipeline Final response, error pages
Map() N/A (branches) Branch pipeline Path-based routing

Use() - Chains Middleware:

app.Use(async (context, next) =>
{
    // Before next middleware
    Console.WriteLine("Request received");
    
    await next(context); // Call next middleware (IMPORTANT!)
    
    // After next middleware
    Console.WriteLine("Response sent");
});

Run() - Terminal Middleware:

app.Run(async context =>
{
    // No next() call - pipeline ends here
    await context.Response.WriteAsync("Hello World!");
    // Any middleware after Run() is NEVER executed
});

Pipeline Execution Order:

// Example Pipeline
app.Use(async (context, next) =>
{
    Console.WriteLine("Middleware 1 - Before");
    await next(context);
    Console.WriteLine("Middleware 1 - After");
});

app.Use(async (context, next) =>
{
    Console.WriteLine("Middleware 2 - Before");
    await next(context);
    Console.WriteLine("Middleware 2 - After");
});

app.Run(async context =>
{
    Console.WriteLine("Terminal Middleware");
    await context.Response.WriteAsync("Response");
});

// Output:
// Middleware 1 - Before
// Middleware 2 - Before
// Terminal Middleware
// Middleware 2 - After
// Middleware 1 - After

Map() - Branch Pipeline:

// Branch based on path
app.Map("/api", apiApp =>
{
    apiApp.Run(async context =>
    {
        await context.Response.WriteAsync("API Response");
    });
});

app.Map("/admin", adminApp =>
{
    adminApp.Run(async context =>
    {
        await context.Response.WriteAsync("Admin Panel");
    });
});

// Default route
app.Run(async context =>
{
    await context.Response.WriteAsync("Home Page");
});

💼 Practical Example

In one of my projects, I accidentally used Run() instead of Use() for a logging middleware. The pipeline stopped after logging, and authentication/authorization never ran—all requests returned "Logged" without checking permissions! The fix was changing app.Run() to app.Use() and calling await next(). Now I always use Run() only for final handlers like 404 pages or health check endpoints. 

// WRONG - Run() stops pipeline
app.Run(async context =>
{
    Console.WriteLine("Logging request");
    // No next() - authentication never runs!
});

app.UseAuthentication(); // NEVER EXECUTED!
app.UseAuthorization(); // NEVER EXECUTED!

// CORRECT - Use() chains middleware
app.Use(async (context, next) =>
{
    Console.WriteLine("Logging request");
    await next(context); // Continue to next middleware
    Console.WriteLine("Request completed");
});

app.UseAuthentication(); // NOW EXECUTES
app.UseAuthorization(); // NOW EXECUTES
app.MapControllers();

// Use Run() only at the end for fallback responses
app.Run(async context =>
{
    context.Response.StatusCode = 404;
    await context.Response.WriteAsync("404 - Not Found");
});

🎯 Key Takeaway

Use() chains middleware by calling await next()—use for logging, auth, request processing. Run() terminates the pipeline without calling next()—use only for final responses like 404 pages or health checks. Never use Run() in the middle of the pipeline. Map() branches pipelines based on request paths.

Q12: What is the difference between IQueryable and IEnumerable?

📚 Theory

IEnumerable<T> executes queries in-memory using LINQ-to-Objects—data is fetched from the database first, then filtered in C#. IQueryable<T> executes queries at the database using LINQ-to-SQL/EF—filters are translated to SQL and executed server-side. IQueryable supports deferred execution (builds expression tree) and query optimization. Use IQueryable for database queries (EF Core), IEnumerable for in-memory collections. Converting IQueryable → IEnumerable (.ToList()) executes the query immediately.

⚙️ How It Works in .NET Core

Key Differences:

Feature IEnumerable<T> IQueryable<T>
Namespace System.Collections System.Linq
Execution In-memory (client-side) Database (server-side)
Query Provider LINQ-to-Objects LINQ-to-SQL/EF
Expression Func<T, bool> Expression<Func<T, bool>>
Deferred Execution Yes Yes (+ optimized)
Performance Slower (fetches all data) Faster (filters at DB)
Use Case In-memory collections Database queries

IEnumerable Example (Bad for DB queries):

// BAD: Fetches ALL orders, then filters in C#
IEnumerable<Order> orders = _context.Orders; // All records loaded to memory!
var result = orders.Where(o => o.Total > 1000); // Filters in C# (slow)

// Generated SQL: SELECT * FROM Orders (all rows)
// Then filters in memory

IQueryable Example (Good for DB queries):

// GOOD: Filters at database
IQueryable<Order> orders = _context.Orders; // No data fetched yet
var result = orders.Where(o => o.Total > 1000); // Builds expression tree
var list = await result.ToListAsync(); // NOW executes

// Generated SQL: SELECT * FROM Orders WHERE Total > 1000 (efficient)

Deferred Execution:

// Query is NOT executed until enumeration
IQueryable<Order> query = _context.Orders
    .Where(o => o.Total > 1000)
    .OrderBy(o => o.OrderDate); // No DB hit yet

// Can add more filters
if (includeShipped)
{
    query = query.Where(o => o.Status == "Shipped"); // Still no DB hit
}

// Executes when enumerated
var orders = await query.ToListAsync(); // NOW hits database

// Final SQL: 
// SELECT * FROM Orders 
// WHERE Total > 1000 AND Status = 'Shipped'
// ORDER BY OrderDate

Converting IQueryable to IEnumerable:

IQueryable<Order> queryable = _context.Orders.Where(o => o.Total > 1000);

// Convert to IEnumerable (executes query immediately)
IEnumerable<Order> enumerable = queryable.ToList(); // DB query executes

// Further filtering happens in-memory (not at DB)
var filtered = enumerable.Where(o => o.Status == "Shipped"); // In-memory filter

💼 Practical Example

In one of my projects, we had a slow API endpoint loading 100,000 orders. The developer used IEnumerable and filtered with .Where() in C#—it loaded all 100K rows into memory, then filtered! I changed it to IQueryable, which generated optimized SQL with WHERE clause at the database. Query time dropped from 30 seconds to 200ms. Lesson: always use IQueryable for EF Core queries, convert to IEnumerable only when needed. 

// BEFORE (Slow) - IEnumerable
public async Task<List<Order>> GetOrdersAsync(decimal minTotal, string status)
{
    // BAD: Loads all orders into memory first!
    IEnumerable<Order> orders = await _context.Orders.ToListAsync(); // 100K rows
    
    // Filters in C# (slow)
    return orders
        .Where(o => o.Total > minTotal)
        .Where(o => o.Status == status)
        .ToList();
}

// AFTER (Fast) - IQueryable
public async Task<List<Order>> GetOrdersAsync(decimal minTotal, string status)
{
    // GOOD: Filters at database
    IQueryable<Order> query = _context.Orders
        .Where(o => o.Total > minTotal)
        .Where(o => o.Status == status);
    
    return await query.ToListAsync(); // Executes optimized SQL
}

// Generated SQL (optimized):
// SELECT * FROM Orders 
// WHERE Total > @minTotal AND Status = @status

🎯 Key Takeaway

IEnumerable executes queries in-memory (C#), IQueryable executes at database (SQL). Always use IQueryable for EF Core database queries to enable server-side filtering and optimization. Convert to IEnumerable (.ToList()) only when needed. IQueryable supports deferred execution—query builds until enumerated. Performance difference is massive for large datasets.

Q13: How do you implement global exception handling in an API?

📚 Theory

Global exception handling centralizes error processing, returning consistent error responses instead of crashing or leaking stack traces. .NET Core provides three approaches: UseExceptionHandler middleware (recommended for production), Exception Filters (MVC-specific), and Custom Exception Middleware (most flexible). Best practices: log exceptions, return standard error format (ProblemDetails), hide sensitive info in production, use appropriate HTTP status codes.

⚙️ How It Works in .NET Core

Three Approaches:

1. UseExceptionHandler Middleware (Simplest):

// Program.cs
var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseDeveloperExceptionPage(); // Detailed errors in dev
}
else
{
    app.UseExceptionHandler("/error"); // Redirect to error endpoint
}

// Error handling endpoint
app.Map("/error", (HttpContext context) =>
{
    var exception = context.Features.Get<IExceptionHandlerFeature>()?.Error;
    
    return Results.Problem(
        title: "An error occurred",
        statusCode: 500,
        detail: exception?.Message);
});

2. Exception Filter (MVC/API Controllers):

// Custom Exception Filter
public class GlobalExceptionFilter : IExceptionFilter
{
    private readonly ILogger<GlobalExceptionFilter> _logger;
    private readonly IHostEnvironment _env;

    public GlobalExceptionFilter(ILogger<GlobalExceptionFilter> logger, IHostEnvironment env)
    {
        _logger = logger;
        _env = env;
    }

    public void OnException(ExceptionContext context)
    {
        // Log exception
        _logger.LogError(context.Exception, "Unhandled exception occurred");

        // Create error response
        var problemDetails = new ProblemDetails
        {
            Status = 500,
            Title = "An error occurred",
            Detail = _env.IsDevelopment() ? context.Exception.Message : "Internal server error"
        };

        context.Result = new ObjectResult(problemDetails)
        {
            StatusCode = 500
        };

        context.ExceptionHandled = true; // Mark as handled
    }
}

// Register globally
builder.Services.AddControllers(options =>
{
    options.Filters.Add<GlobalExceptionFilter>();
});

3. Custom Exception Middleware (Most Flexible):

// Custom Exception Middleware
public class ExceptionHandlingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ExceptionHandlingMiddleware> _logger;

    public ExceptionHandlingMiddleware(RequestDelegate next, ILogger<ExceptionHandlingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context); // Call next middleware
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Unhandled exception: {Message}", ex.Message);
            await HandleExceptionAsync(context, ex);
        }
    }

    private static async Task HandleExceptionAsync(HttpContext context, Exception exception)
    {
        context.Response.ContentType = "application/json";
        
        // Custom status codes based on exception type
        context.Response.StatusCode = exception switch
        {
            NotFoundException => 404,
            UnauthorizedException => 401,
            ValidationException => 400,
            _ => 500
        };

        var response = new
        {
            status = context.Response.StatusCode,
            message = exception.Message,
            timestamp = DateTime.UtcNow
        };

        await context.Response.WriteAsJsonAsync(response);
    }
}

// Extension method
public static class ExceptionMiddlewareExtensions
{
    public static IApplicationBuilder UseGlobalExceptionHandler(this IApplicationBuilder app)
    {
        return app.UseMiddleware<ExceptionHandlingMiddleware>();
    }
}

// Usage in Program.cs
app.UseGlobalExceptionHandler(); // Add at the top of pipeline
app.UseRouting();
app.MapControllers();

💼 Practical Example

In one of my projects, we had exceptions exposing sensitive data (SQL connection strings, stack traces) to clients in production. I implemented custom exception middleware that logged full details (including stack traces) to Azure Application Insights but returned generic messages to clients. For known exceptions (NotFoundException, ValidationException), we returned appropriate status codes (404, 400). For unexpected errors, we returned 500 with a correlation ID for debugging. This reduced support tickets by 40% because users got meaningful error messages. 

// Production-Ready Exception Middleware
public class ProductionExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ProductionExceptionMiddleware> _logger;

    public ProductionExceptionMiddleware(RequestDelegate next, ILogger<ProductionExceptionMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (Exception ex)
        {
            var correlationId = Guid.NewGuid().ToString();
            
            // Log full details with correlation ID
            _logger.LogError(ex, 
                "[CorrelationId: {CorrelationId}] Unhandled exception: {Message}. StackTrace: {StackTrace}",
                correlationId, ex.Message, ex.StackTrace);

            // Return safe error to client
            context.Response.ContentType = "application/json";
            context.Response.StatusCode = ex switch
            {
                NotFoundException => 404,
                UnauthorizedException => 401,
                ValidationException => 400,
                _ => 500
            };

            var response = new
            {
                status = context.Response.StatusCode,
                message = ex is ValidationException ? ex.Message : "An error occurred",
                correlationId = correlationId, // For support/debugging
                timestamp = DateTime.UtcNow
            };

            await context.Response.WriteAsJsonAsync(response);
        }
    }
}

// Usage
app.UseMiddleware<ProductionExceptionMiddleware>();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();

🎯 Key Takeaway

Implement global exception handling using UseExceptionHandler (simple), Exception Filters (MVC-only), or Custom Middleware (flexible). Always log full exception details but return safe messages to clients. Use ProblemDetails for standard error format. Return appropriate HTTP status codes (404, 400, 401, 500). Include correlation IDs for debugging. Never expose stack traces in production.
