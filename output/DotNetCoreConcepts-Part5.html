Q27. How to add Swagger for documentation of API?
📚 Theory
Swagger (now OpenAPI) generates interactive API documentation automatically by inspecting your controllers, models, and endpoints. It provides a web UI where developers can test endpoints, view request/response schemas, and understand authentication requirements. In .NET Core, Swashbuckle.AspNetCore is the standard library that implements OpenAPI specification, generating JSON documentation and hosting the Swagger UI.
⚙️ How it works in .NET Core
  • Swashbuckle.AspNetCore NuGet package (contains SwaggerGen + SwaggerUI)
  • AddSwaggerGen() registers the Swagger generator middleware
  • UseSwagger() serves the OpenAPI JSON at /swagger/v1/swagger.json
  • UseSwaggerUI() serves the interactive UI at /swagger
  • XML comments from code are included for richer documentation
  • Annotations like [ProducesResponseType], [SwaggerOperation] enhance metadata
💼 Practical Example
When migrating our legacy SOAP services to REST APIs, I added Swagger to help the front-end team understand the new endpoints without lengthy documentation meetings: 
// 1. Install NuGet package
// dotnet add package Swashbuckle.AspNetCore

// 2. Program.cs - Configure Swagger
var builder = WebApplication.CreateBuilder(args);

// Add services
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();

// Configure Swagger with XML comments
builder.Services.AddSwaggerGen(options =>
{
    options.SwaggerDoc("v1", new OpenApiInfo
    {
        Title = "MIC API",
        Version = "v1",
        Description = "API for binder management and project approval",
        Contact = new OpenApiContact
        {
            Name = "Support Team",
            Email = "support@example.com"
        }
    });

    // Enable XML comments for documentation
    var xmlFile = $"{Assembly.GetExecutingAssembly().GetName().Name}.xml";
    var xmlPath = Path.Combine(AppContext.BaseDirectory, xmlFile);
    options.IncludeXmlComments(xmlPath);

    // Add JWT authentication to Swagger
    options.AddSecurityDefinition("Bearer", new OpenApiSecurityScheme
    {
        Description = "JWT Authorization header using Bearer scheme",
        Name = "Authorization",
        In = ParameterLocation.Header,
        Type = SecuritySchemeType.ApiKey,
        Scheme = "Bearer"
    });

    options.AddSecurityRequirement(new OpenApiSecurityRequirement
    {
        {
            new OpenApiSecurityScheme
            {
                Reference = new OpenApiReference
                {
                    Type = ReferenceType.SecurityScheme,
                    Id = "Bearer"
                }
            },
            new string[] { }
        }
    });
});

var app = builder.Build();

// Enable Swagger in Development and Staging
if (app.Environment.IsDevelopment() || app.Environment.IsStaging())
{
    app.UseSwagger();
    app.UseSwaggerUI(c =>
    {
        c.SwaggerEndpoint("/swagger/v1/swagger.json", "MIC API v1");
        c.RoutePrefix = "swagger"; // Access at /swagger
    });
}

app.UseAuthorization();
app.MapControllers();
app.Run();

// 3. Enable XML documentation in .csproj
// <PropertyGroup>
//   <GenerateDocumentationFile>true</GenerateDocumentationFile>
//   <NoWarn>$(NoWarn);1591</NoWarn>
// </PropertyGroup>

// 4. Document your API endpoints
[ApiController]
[Route("api/[controller]")]
public class BindersController : ControllerBase
{
    /// <summary>
    /// Gets a binder by its ID
    /// </summary>
    /// <param name="id">The binder ID</param>
    /// <returns>Binder details</returns>
    /// <response code="200">Returns the binder</response>
    /// <response code="404">Binder not found</response>
    [HttpGet("{id}")]
    [ProducesResponseType(StatusCodes.Status200OK)]
    [ProducesResponseType(StatusCodes.Status404NotFound)]
    public async Task<ActionResult<BinderDto>> GetBinder(int id)
    {
        // Implementation...
    }
}
After this setup, our front-end team could access /swagger and test every endpoint directly from the browser. It cut our API onboarding time by 70%.
🎯 Key Takeaway
Use Swashbuckle.AspNetCore package. Configure with AddSwaggerGen(), enable with UseSwagger() and UseSwaggerUI(). Use XML comments and [ProducesResponseType] attributes for rich documentation. Disable in production for security unless behind authentication.

Q28. Routing - Understand attribute vs conventional routing
📚 Theory
Conventional routing defines URL patterns globally in one place (Program.cs) using MapControllerRoute(), following patterns like {controller}/{action}/{id}. Attribute routing defines routes directly on controllers/actions using attributes like [Route], [HttpGet], giving fine-grained control per endpoint. APIs typically use attribute routing (RESTful), while MVC views often use conventional routing (consistency across many pages).
⚙️ How it works in .NET Core
  • Conventional: MapControllerRoute() in Program.cs defines global patterns
  • Attribute: [Route], [HttpGet], [HttpPost] on controllers/actions
  • Precedence: Attribute routing overrides conventional routing
  • Token replacement: [controller], [action] tokens in attribute routes
  • Route constraints: {id:int}, {slug:minlength(3)} for validation
  • Order matters: Use [Route(Order = 1)] when routes might conflict
💼 Practical Example
In our migration from WebForms to .NET Core, I used conventional routing for admin MVC pages (consistent structure) and attribute routing for RESTful APIs: 
// ===== CONVENTIONAL ROUTING (Program.cs) =====
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllersWithViews();

var app = builder.Build();

// Define conventional routes
app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}"
);

// Creates URLs like:
// /Home/Index
// /Binder/Details/123
// /Project/Approval

// ===== ATTRIBUTE ROUTING (API Controller) =====
[ApiController]
[Route("api/[controller]")] // Token replaced with "binders"
public class BindersController : ControllerBase
{
    // GET: api/binders
    [HttpGet]
    public IActionResult GetAll() { /* ... */ }

    // GET: api/binders/123
    [HttpGet("{id:int}")] // Route constraint ensures int
    public IActionResult GetById(int id) { /* ... */ }

    // POST: api/binders
    [HttpPost]
    public IActionResult Create([FromBody] BinderDto dto) { /* ... */ }

    // GET: api/binders/search?query=test
    [HttpGet("search")] // Custom segment
    public IActionResult Search([FromQuery] string query) { /* ... */ }

    // GET: api/binders/123/documents
    [HttpGet("{id:int}/documents")]
    public IActionResult GetDocuments(int id) { /* ... */ }
}

// ===== MIX OF BOTH (MVC Controller) =====
// Uses conventional routing by default, but can override with attributes
public class BinderController : Controller
{
    // Uses conventional: /Binder/Index
    public IActionResult Index() { /* ... */ }

    // Override with attribute: /binder-detail/123
    [Route("binder-detail/{id:int}")]
    public IActionResult Details(int id) { /* ... */ }
}

// ===== REAL SCENARIO I HIT =====
// We had a route collision:
// /api/binders/active (should get active binders)
// /api/binders/{id} (id="active" was interpreted as string!)

// Solution: Use route order and constraints
[HttpGet("active", Order = 1)] // Matches first
public IActionResult GetActive() { /* ... */ }

[HttpGet("{id:int}", Order = 2)] // Only matches if id is int
public IActionResult GetById(int id) { /* ... */ }
🎯 Key Takeaway
Conventional routing: Good for MVC apps with consistent patterns (admin dashboards). Attribute routing: Preferred for REST APIs, gives precise control per endpoint. Use route constraints ({id:int}) to avoid ambiguity. APIs should always use attribute routing with [ApiController].

Q29. Logging - Use built-in logging
📚 Theory
.NET Core includes a built-in logging framework that supports multiple providers (Console, Debug, EventSource, etc.) and integrates with third-party loggers like Serilog or NLog. The ILogger<T> interface provides methods for different log levels (Trace, Debug, Info, Warning, Error, Critical). Logging configuration is managed through appsettings.json, allowing different log levels per environment without code changes.
⚙️ How it works in .NET Core
  • ILogger<T> injected via DI, T is the category (usually class name)
  • ILoggerFactory creates logger instances (rarely used directly)
  • Log levels: Trace (0) → Debug (1) → Information (2) → Warning (3) → Error (4) → Critical (5)
  • Providers: Console, Debug, EventSource, EventLog (Windows), AzureAppServices
  • Configuration: appsettings.json "Logging" section defines levels per namespace
  • Structured logging: Use message templates with parameters for better querying
💼 Practical Example
In our microservices project, logging was critical for debugging production issues. Here's how I set it up: 
// ===== appsettings.json - Configuration =====
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning", // Reduce noise from framework
      "Microsoft.EntityFrameworkCore": "Information", // See SQL queries
      "MyApp.Services": "Debug" // Detailed logs for my services
    }
  }
}

// ===== appsettings.Development.json =====
{
  "Logging": {
    "LogLevel": {
      "Default": "Debug" // More verbose in dev
    }
  }
}

// ===== Program.cs - Add logging providers =====
var builder = WebApplication.CreateBuilder(args);

// Built-in logging is added by default, but you can customize:
builder.Logging.ClearProviders(); // Remove defaults
builder.Logging.AddConsole(); // Console output
builder.Logging.AddDebug(); // Debug window in VS
builder.Logging.AddEventSourceLogger(); // For ETW tracing

// Optional: Add Serilog for production
// builder.Host.UseSerilog((context, config) => 
//     config.ReadFrom.Configuration(context.Configuration));

// ===== Using ILogger in a Controller =====
[ApiController]
[Route("api/[controller]")]
public class BindersController : ControllerBase
{
    private readonly ILogger<BindersController> _logger;
    private readonly IBinderService _binderService;

    public BindersController(
        ILogger<BindersController> logger, 
        IBinderService binderService)
    {
        _logger = logger;
        _binderService = binderService;
    }

    [HttpGet("{id}")]
    public async Task<IActionResult> GetBinder(int id)
    {
        // Structured logging - parameters are captured separately
        _logger.LogInformation("Fetching binder with ID: {BinderId}", id);

        try
        {
            var binder = await _binderService.GetByIdAsync(id);

            if (binder == null)
            {
                _logger.LogWarning("Binder {BinderId} not found", id);
                return NotFound();
            }

            _logger.LogDebug("Successfully retrieved binder {BinderId} with {DocumentCount} documents", 
                id, binder.Documents.Count);

            return Ok(binder);
        }
        catch (DbException dbEx)
        {
            // Error with exception details
            _logger.LogError(dbEx, 
                "Database error retrieving binder {BinderId}", id);
            return StatusCode(500, "Database error occurred");
        }
        catch (Exception ex)
        {
            // Critical error - requires immediate attention
            _logger.LogCritical(ex, 
                "Critical error in GetBinder for ID {BinderId}", id);
            return StatusCode(500);
        }
    }
}

// ===== Using ILogger in a Service =====
public class BinderService : IBinderService
{
    private readonly ILogger<BinderService> _logger;

    public BinderService(ILogger<BinderService> logger)
    {
        _logger = logger;
    }

    public async Task ProcessBinderAsync(int id)
    {
        // Using LoggerMessage for high-performance logging
        using (_logger.BeginScope("Processing binder {BinderId}", id))
        {
            _logger.LogInformation("Starting binder processing");
            
            // Scoped logs will include BinderId in all messages
            await ProcessSteps();
            
            _logger.LogInformation("Completed binder processing");
        }
    }
}

// REAL PRODUCTION TIP:
// We integrated with Azure Application Insights:
builder.Services.AddApplicationInsightsTelemetry();
// All ILogger calls automatically went to App Insights!
🎯 Key Takeaway
Inject ILogger<T> via DI, use structured logging with message templates. Configure log levels in appsettings.json per namespace. Use appropriate levels (Debug for dev, Information for normal flow, Error for exceptions). Integrate with Application Insights or Serilog for production.

Q30. Controllers - Build simple controllers
📚 Theory
Controllers handle HTTP requests and return responses. In .NET Core, API controllers inherit from ControllerBase and return data (JSON/XML), while MVC controllers inherit from Controller and return views. Controllers contain action methods decorated with HTTP verb attributes ([HttpGet], [HttpPost], etc.). The [ApiController] attribute enables automatic model validation, binding from body by default, and problem details for errors.
⚙️ How it works in .NET Core
  • ControllerBase - Base for API controllers (no view support)
  • Controller - Base for MVC controllers (includes view support)
  • [ApiController] - Enables automatic 400 responses, attribute routing required
  • [Route] - Defines route template for controller/actions
  • Action methods - Return IActionResult, ActionResult<T>, or specific types
  • Model binding - [FromBody], [FromQuery], [FromRoute], [FromForm]
💼 Practical Example
When building our first .NET Core API for the binder management system, I started with these simple controllers: 
// ===== SIMPLE API CONTROLLER =====
[ApiController] // Enables automatic features
[Route("api/[controller]")] // Maps to api/binders
public class BindersController : ControllerBase
{
    private readonly IBinderService _binderService;
    private readonly ILogger<BindersController> _logger;

    // Dependency injection in constructor
    public BindersController(
        IBinderService binderService, 
        ILogger<BindersController> logger)
    {
        _binderService = binderService;
        _logger = logger;
    }

    // GET: api/binders
    [HttpGet]
    public async Task<ActionResult<IEnumerable<BinderDto>>> GetAll()
    {
        var binders = await _binderService.GetAllAsync();
        return Ok(binders); // Returns 200 with data
    }

    // GET: api/binders/123
    [HttpGet("{id:int}")] // Route parameter constraint
    public async Task<ActionResult<BinderDto>> GetById(int id)
    {
        var binder = await _binderService.GetByIdAsync(id);
        
        if (binder == null)
            return NotFound(); // Returns 404

        return Ok(binder); // Returns 200 with binder data
    }

    // POST: api/binders
    [HttpPost]
    public async Task<ActionResult<BinderDto>> Create([FromBody] CreateBinderDto dto)
    {
        // [ApiController] automatically validates ModelState
        // No need for manual ModelState.IsValid check!

        var binder = await _binderService.CreateAsync(dto);
        
        // Returns 201 with Location header
        return CreatedAtAction(
            nameof(GetById), 
            new { id = binder.Id }, 
            binder
        );
    }

    // PUT: api/binders/123
    [HttpPut("{id:int}")]
    public async Task<IActionResult> Update(int id, [FromBody] UpdateBinderDto dto)
    {
        if (id != dto.Id)
            return BadRequest("ID mismatch"); // Returns 400

        var updated = await _binderService.UpdateAsync(dto);
        
        if (!updated)
            return NotFound();

        return NoContent(); // Returns 204
    }

    // DELETE: api/binders/123
    [HttpDelete("{id:int}")]
    public async Task<IActionResult> Delete(int id)
    {
        var deleted = await _binderService.DeleteAsync(id);
        
        if (!deleted)
            return NotFound();

        return NoContent(); // Returns 204
    }

    // GET: api/binders/search?query=test&pageSize=20
    [HttpGet("search")]
    public async Task<ActionResult<PagedResult<BinderDto>>> Search(
        [FromQuery] string query, 
        [FromQuery] int pageSize = 10)
    {
        var results = await _binderService.SearchAsync(query, pageSize);
        return Ok(results);
    }
}

// ===== DTO CLASSES =====
public class BinderDto
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string Description { get; set; }
    public DateTime CreatedDate { get; set; }
}

public class CreateBinderDto
{
    [Required]
    [StringLength(100)]
    public string Name { get; set; }

    [MaxLength(500)]
    public string Description { get; set; }
}

// ===== SIMPLE MVC CONTROLLER (for Views) =====
public class HomeController : Controller // Inherits Controller, not ControllerBase
{
    // GET: /Home/Index
    public IActionResult Index()
    {
        return View(); // Returns a Razor view
    }

    // POST: /Home/Submit
    [HttpPost]
    public IActionResult Submit([FromForm] BinderViewModel model)
    {
        if (!ModelState.IsValid)
            return View(model);

        // Process form...
        
        return RedirectToAction(nameof(Index));
    }
}

// ===== REGISTER IN PROGRAM.CS =====
builder.Services.AddControllers(); // For API controllers
// OR
builder.Services.AddControllersWithViews(); // For MVC controllers with views

app.MapControllers(); // Map controller routes
This controller pattern became our standard across all microservices. Clean, testable, and followed REST conventions.
🎯 Key Takeaway
API controllers inherit from ControllerBase and use [ApiController] attribute. Use HTTP verb attributes ([HttpGet], [HttpPost]) with route templates. Return ActionResult<T> for type safety. Use appropriate status codes (200, 201, 204, 400, 404). Inject dependencies via constructor.
