Q79: Add, Attach and Update of EF Core

📚 Theory

EF Core tracks entity state changes using the ChangeTracker. Add() marks an entity as Added (for new records), Attach() marks it as Unchanged (useful for disconnected scenarios where you want to track an existing entity without querying), and Update() marks it as Modified (tells EF Core the entity has changed and should be updated in the database). These methods control how EF Core generates SQL INSERT, UPDATE, or does nothing during SaveChanges().

⚙️ How it Works in .NET Core

• Add() → EntityState.Added → Generates INSERT
• Attach() → EntityState.Unchanged → No SQL until you modify properties
• Update() → EntityState.Modified → Generates UPDATE for all columns
Use DbContext.Entry(entity).State to manually inspect or set entity state. In disconnected scenarios (like Web APIs), Attach() is crucial before modifying specific properties.

💼 Practical Example

In one of my projects, we had a REST API where clients sent updated employee data without fetching it first. I used Attach() to tell EF Core the entity exists, then marked specific properties as modified to avoid updating the entire row. 

// Client sends updated employee (disconnected)
public async Task UpdateEmployeeSalary(Employee employee)
{
    // Attach marks as Unchanged - no query to DB
    _context.Attach(employee);
    
    // Mark only Salary as modified
    _context.Entry(employee).Property(e => e.Salary).IsModified = true;
    
    await _context.SaveChangesAsync();
    // Generates: UPDATE Employees SET Salary = @p0 WHERE Id = @p1
}

// VS using Update() which updates ALL columns
public async Task UpdateEmployeeFull(Employee employee)
{
    _context.Update(employee); // Marks entire entity as Modified
    await _context.SaveChangesAsync();
    // Generates: UPDATE Employees SET Name = @p0, Salary = @p1, Dept = @p2... WHERE Id = @pN
}

🎯 Key Takeaway

Use Add() for new entities, Attach() + property-level modifications for efficient disconnected updates, and Update() when you're sure the entire entity has changed. Understanding entity state management prevents unnecessary database round-trips and optimizes SQL generation.

Q80: LINQ query to get employees having salary greater than avg salary of the group

📚 Theory

This is a classic grouped aggregation problem where you need to calculate the average salary per group (e.g., by Department), then filter employees whose salary exceeds their group's average. In LINQ, you use GroupBy() to create groups, Average() to compute the mean, and SelectMany() or nested queries to flatten results and apply the filter.

⚙️ How it Works in .NET Core

LINQ provides two syntaxes: query syntax (from...where...select) and method syntax (lambda-based). For this problem, method syntax with GroupBy(), Where(), and SelectMany() is cleaner. If using EF Core, this translates to SQL with subqueries or window functions (depending on provider support).

💼 Practical Example

In a payroll system I worked on, we needed to identify high performers (above-average earners) per department for bonus calculations. Here's how I solved it with LINQ: 

public class Employee
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string Department { get; set; }
    public decimal Salary { get; set; }
}

// Method 1: Using GroupBy + SelectMany
var result = employees
    .GroupBy(e => e.Department)
    .SelectMany(group => 
    {
        var avgSalary = group.Average(e => e.Salary);
        return group.Where(e => e.Salary > avgSalary);
    });

// Method 2: Query Syntax (more readable for some)
var result2 = from emp in employees
                group emp by emp.Department into deptGroup
                let avgSalary = deptGroup.Average(e => e.Salary)
                from emp in deptGroup
                where emp.Salary > avgSalary
                select emp;

// Method 3: Join with grouped averages (EF Core friendly)
var avgSalaries = employees
    .GroupBy(e => e.Department)
    .Select(g => new { Department = g.Key, AvgSalary = g.Average(e => e.Salary) });

var result3 = employees
    .Join(avgSalaries, 
        e => e.Department, 
        avg => avg.Department,
        (e, avg) => new { Employee = e, avg.AvgSalary })
    .Where(x => x.Employee.Salary > x.AvgSalary)
    .Select(x => x.Employee);

🎯 Key Takeaway

Use GroupBy() + SelectMany() for in-memory collections or Join() with pre-computed averages for better EF Core SQL generation. This pattern shows mastery of LINQ aggregation and grouping—critical for data processing tasks.

Q81: Can you explain lambda expressions with example?

📚 Theory

Lambda expressions are anonymous functions written using the => (lambda operator). They're syntactic sugar for delegates and are heavily used with LINQ, event handlers, and functional programming. The format is (parameters) => expression or (parameters) => { statements }. Lambdas can capture variables from the enclosing scope (closures), making them powerful but requiring careful memory management.

⚙️ How it Works in .NET Core

Lambdas compile to delegates like Func<T, TResult>, Action<T>, or Predicate<T>. The compiler infers types using type inference. Common uses:
• LINQ: .Where(x => x.Age > 18)
• Events: button.Click += (s, e) => Console.WriteLine("Clicked");
• Async: await Task.Run(() => DoWork());
Expression-bodied members (int Square(int x) => x * x;) are a related feature.

💼 Practical Example

In a microservice I built, we used lambdas extensively for filtering, validation, and background tasks. Here are real scenarios: 

// 1. LINQ filtering (most common)
var activeUsers = users.Where(u => u.IsActive && u.LastLogin > DateTime.Now.AddDays(-30));

// 2. Func delegate for reusable logic
Func<int, int, int> add = (x, y) => x + y;
int result = add(5, 3); // 8

// 3. Multi-line lambda with block body
Action<string> processOrder = (orderId) =>
{
    _logger.LogInformation($"Processing {orderId}");
    var order = _repo.GetOrder(orderId);
    order.Ship();
};

// 4. Closure (capturing variable from outer scope)
int threshold = 100;
var expensive = products.Where(p => p.Price > threshold); // threshold captured

// 5. Expression-bodied member (not technically lambda, but similar)
public class Product
{
    public decimal Price { get; set; }
    public decimal TaxAmount => Price * 0.08m; // Expression body
}

// 6. Async lambda in Task.Run
await Task.Run(async () => 
{
    await Task.Delay(1000);
    return "Done";
});

🎯 Key Takeaway

Lambdas are essential for modern C# development, especially with LINQ and async programming. They improve code readability and reduce boilerplate. Understanding closures and delegate types (Func, Action) is key to using them effectively.

Q82: Write the code to count frequency of the char in a string

📚 Theory

Counting character frequency is a common string manipulation problem. You iterate through the string and maintain a count for each character. Common approaches: Dictionary<char, int> for manual counting, or LINQ's GroupBy() for a functional approach. Edge cases include case sensitivity, whitespace handling, and Unicode characters.

⚙️ How it Works in .NET Core

.NET provides multiple ways:
• Dictionary approach: Loop through string, use TryGetValue() or ContainsKey()
• LINQ GroupBy: str.GroupBy(c => c).ToDictionary(g => g.Key, g => g.Count())
• LINQ aggregate: More advanced with Aggregate()
For performance-critical code, Dictionary is faster than LINQ due to fewer allocations.

💼 Practical Example

I used this pattern in a text analytics feature where we analyzed customer feedback. Here are 3 implementations I've used in production: 

// Method 1: Dictionary (most efficient)
public static Dictionary<char, int> CountCharacterFrequency(string input)
{
    var frequency = new Dictionary<char, int>();
    
    foreach (char c in input)
    {
        if (frequency.ContainsKey(c))
            frequency[c]++;
        else
            frequency[c] = 1;
    }
    
    return frequency;
}

// Method 2: Dictionary with TryGetValue (cleaner)
public static Dictionary<char, int> CountCharFrequency2(string input)
{
    var frequency = new Dictionary<char, int>();
    
    foreach (char c in input)
    {
        frequency.TryGetValue(c, out int count);
        frequency[c] = count + 1;
    }
    
    return frequency;
}

// Method 3: LINQ GroupBy (most readable, slightly slower)
public static Dictionary<char, int> CountCharFrequency3(string input)
{
    return input
        .GroupBy(c => c)
        .ToDictionary(g => g.Key, g => g.Count());
}

// Method 4: Case-insensitive with whitespace filtering
public static Dictionary<char, int> CountCharFrequency4(string input)
{
    return input
        .Where(c => !Char.IsWhiteSpace(c))
        .Select(c => Char.ToLower(c))
        .GroupBy(c => c)
        .ToDictionary(g => g.Key, g => g.Count());
}

// Usage example
string text = "hello world";
var result = CountCharacterFrequency(text);

foreach (var kvp in result)
{
    Console.WriteLine($"{kvp.Key}: {kvp.Value}");
}
// Output: h:1, e:1, l:3, o:2, (space):1, w:1, r:1, d:1

🎯 Key Takeaway

For interviews, show multiple approaches. Dictionary with TryGetValue() is efficient and production-ready. LINQ GroupBy() demonstrates functional programming knowledge. Mention edge cases like case sensitivity and special characters.

Q83: How to write raw query in Entity Framework?

📚 Theory

Sometimes LINQ doesn't generate optimal SQL or you need database-specific features (stored procedures, window functions, CTEs). EF Core provides raw SQL execution through FromSqlRaw() and FromSqlInterpolated() for queries, and ExecuteSqlRaw()/ExecuteSqlInterpolated() for non-query commands (INSERT/UPDATE/DELETE). Always use parameterized queries to prevent SQL injection.

⚙️ How it Works in .NET Core

EF Core 5+ methods:
• FromSqlRaw/FromSqlInterpolated: Compose-able queries returning entities (can add .Where(), .OrderBy() after)
• ExecuteSqlRaw/ExecuteSqlInterpolated: Non-query commands returning affected row count
• FromSqlRaw: Use with string.Format-style placeholders ({0}, {1})
• FromSqlInterpolated: Use with string interpolation ($"...{param}")—automatically parameterized
For stored procedures, use FromSqlRaw("EXEC ProcName @p0", param)

💼 Practical Example

In a reporting system, I used raw SQL for complex analytics queries that EF Core couldn't optimize. Here's how I handled it: 

// 1. FromSqlInterpolated - Query entities (SAFE - auto-parameterized)
public async Task<List<Employee>> GetEmployeesBySalary(decimal minSalary)
{
    return await _context.Employees
        .FromSqlInterpolated($"SELECT * FROM Employees WHERE Salary > {minSalary}")
        .ToListAsync();
}

// 2. FromSqlRaw with parameters (SAFE - manual parameterization)
public async Task<List<Employee>> GetEmployeesByDept(string dept)
{
    return await _context.Employees
        .FromSqlRaw("SELECT * FROM Employees WHERE Department = {0}", dept)
        .ToListAsync();
}

// 3. Composable query - can chain LINQ after raw SQL
public async Task<List<Employee>> GetTopEarners()
{
    return await _context.Employees
        .FromSqlInterpolated($"SELECT * FROM Employees WHERE IsActive = 1")
        .OrderByDescending(e => e.Salary)
        .Take(10)
        .ToListAsync();
}

// 4. ExecuteSqlRaw - for non-query commands (INSERT/UPDATE/DELETE)
public async Task<int> GiveBonusToDept(string dept, decimal bonus)
{
    return await _context.Database
        .ExecuteSqlInterpolatedAsync(
            $"UPDATE Employees SET Salary = Salary + {bonus} WHERE Department = {dept}");
    // Returns number of affected rows
}

// 5. Stored Procedure execution
public async Task<List<Employee>> GetEmployeesByStoredProc(int departmentId)
{
    return await _context.Employees
        .FromSqlInterpolated($"EXEC GetEmployeesByDepartment {departmentId}")
        .ToListAsync();
}

// 6. DANGER ⚠️ - Never do this (SQL injection vulnerability)
public async Task<List<Employee>> UNSAFE_GetEmployees(string userInput)
{
    // DON'T: String concatenation = SQL INJECTION RISK!
    return await _context.Employees
        .FromSqlRaw($"SELECT * FROM Employees WHERE Name = '{userInput}'")
        .ToListAsync();
}

🎯 Key Takeaway

Use FromSqlInterpolated() for queries and ExecuteSqlInterpolated() for commands—they're safe from SQL injection. Raw SQL is useful for performance optimization and database-specific features, but always validate the generated SQL and use parameters.

Q84: What are different Entity Framework approaches?

📚 Theory

Entity Framework supports three development approaches: Database First (generate models from existing DB), Code First (create DB from C# classes), and Model First (design via visual designer—deprecated in EF Core). Code First is dominant in modern .NET Core development due to version control friendliness, testability, and migration support. Database First is used when working with legacy databases or DBA-controlled schemas.

⚙️ How it Works in .NET Core

Code First (Recommended in EF Core):
• Define POCOs (entities) → Configure via Fluent API or Data Annotations → Generate DB via Migrations
• Commands: dotnet ef migrations add InitialCreate, dotnet ef database update
• Full control over schema, works great with Git, supports CI/CD

Database First (EF Core 5+):
• Reverse engineer existing DB → Generate DbContext and entities via scaffolding
• Command: dotnet ef dbcontext scaffold "ConnectionString" Microsoft.EntityFrameworkCore.SqlServer
• Useful for brownfield projects, but changes require re-scaffolding

Model First:
• Visual designer in VS (EF6 only, NOT supported in EF Core)
• Generate DB from .edmx file—legacy approach, avoid in new projects

💼 Practical Example

I've used both approaches depending on the project. Here's how I applied them: 

// ========== CODE FIRST (My go-to for greenfield projects) ==========

// 1. Define entity
public class Order
{
    public int Id { get; set; }
    public string CustomerName { get; set; }
    public decimal TotalAmount { get; set; }
    public List<OrderItem> Items { get; set; }
}

// 2. Configure in DbContext
public class AppDbContext : DbContext
{
    public DbSet<Order> Orders { get; set; }
    
    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Order>()
            .Property(o => o.CustomerName)
            .IsRequired()
            .HasMaxLength(200);
            
        modelBuilder.Entity<Order>()
            .HasMany(o => o.Items)
            .WithOne()
            .OnDelete(DeleteBehavior.Cascade);
    }
}

// 3. Generate migration
// Terminal: dotnet ef migrations add CreateOrderTable
// Terminal: dotnet ef database update

// ========== DATABASE FIRST (Used in a legacy system migration) ==========

// Scaffold from existing database
// Terminal command I used:
/*
dotnet ef dbcontext scaffold 
    "Server=.;Database=LegacyDB;Trusted_Connection=True;" 
    Microsoft.EntityFrameworkCore.SqlServer 
    --output-dir Models 
    --context-dir Data 
    --context LegacyDbContext
*/

// Generated DbContext (auto-generated, don't modify directly)
public partial class LegacyDbContext : DbContext
{
    public DbSet<Customer> Customers { get; set; }
    public DbSet<Order> Orders { get; set; }
    
    // Auto-generated configuration...
}

// Pro tip: Extend generated context using partial classes
public partial class LegacyDbContext
{
    // Add custom methods here - safe from re-scaffolding
    public async Task<List<Order>> GetRecentOrders()
    {
        return await Orders
            .Where(o => o.OrderDate > DateTime.Now.AddDays(-30))
            .ToListAsync();
    }
}

🎯 Key Takeaway

Code First is the modern standard for EF Core—it's version control friendly, supports migrations, and works well in Agile teams. Database First is for legacy systems or when DBAs own schema design. Avoid Model First in EF Core (not supported). Knowing both approaches shows versatility in brownfield vs. greenfield projects.
