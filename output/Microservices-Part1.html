Q49: Explain the microservices concept
1. Theory
Microservices is an architectural pattern where an application is decomposed into small, independent services that run as separate processes and communicate over lightweight protocols (typically HTTP/REST or messaging). Each microservice owns its data, can be deployed independently, and is organized around specific business capabilities. This contrasts with monolithic architecture where all functionality exists in a single deployable unit. The pattern emphasizes loose coupling, high cohesion, and autonomous teams owning individual services.
2. How it Works in .NET Core
In .NET, we build microservices using ASP.NET Core Web API or gRPC templates. Key components include:
  • ASP.NET Core Minimal APIs or Controllers - for REST endpoints
  • HttpClientFactory - for inter-service communication
  • MassTransit/NServiceBus/RabbitMQ - for async messaging
  • Entity Framework Core - with separate databases per service
  • Docker + Kubernetes - for containerization and orchestration
  • API Gateway - using Ocelot, YARP, or Azure API Management
  • Service Discovery - Consul, Eureka, or built-in Kubernetes DNS
3. Practical Example
In one of my projects, we migrated a monolithic e-commerce platform to microservices. We split it into: OrderService, InventoryService, PaymentService, and NotificationService. Each service had its own PostgreSQL database and was deployed in separate Docker containers on AKS (Azure Kubernetes Service). 
// OrderService - Program.cs (.NET 10)
var builder = WebApplication.CreateBuilder(args);

// Register HttpClient for calling InventoryService
builder.Services.AddHttpClient<IInventoryClient, InventoryClient>(client =>
{
    client.BaseAddress = new Uri(builder.Configuration["Services:Inventory:Url"]);
});

// Register MassTransit for async communication
builder.Services.AddMassTransit(x =>
{
    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host(builder.Configuration["RabbitMQ:Host"]);
    });
});

builder.Services.AddDbContext<OrderDbContext>(options =>
    options.UseNpgsql(builder.Configuration.GetConnectionString("OrderDb")));

var app = builder.Build();

// Minimal API endpoint
app.MapPost("/api/orders", async (
    CreateOrderRequest request,
    OrderDbContext db,
    IInventoryClient inventoryClient,
    IPublishEndpoint publisher) =>
{
    // Check inventory via synchronous HTTP call
    var available = await inventoryClient.CheckAvailabilityAsync(request.ProductId, request.Quantity);
    
    if (!available)
        return Results.BadRequest("Insufficient inventory");

    var order = new Order
    {
        ProductId = request.ProductId,
        Quantity = request.Quantity,
        Status = "Pending"
    };

    db.Orders.Add(order);
    await db.SaveChangesAsync();

    // Publish event asynchronously to NotificationService
    await publisher.Publish(new OrderCreatedEvent { OrderId = order.Id });

    return Results.Created($"/api/orders/{order.Id}", order);
});

app.Run();
The key lesson learned: data consistency across services is hard. We implemented the Saga pattern using MassTransit to handle distributed transactions. When an order failed at the payment stage, we'd publish compensating events to roll back inventory reservations.
4. Key Takeaway
Microservices enable independent deployment and scaling but introduce complexity in distributed systems—focus on clear service boundaries, robust inter-service communication, and eventual consistency patterns. You trade operational complexity for development agility.

Q45: Explain service registry in Service Discovery
1. Theory
A service registry is a centralized database that maintains the network locations (IP addresses and ports) of all available service instances in a microservices architecture. Services register themselves on startup and deregister on shutdown. Client services query the registry to discover available instances before making calls. This enables dynamic scaling since new instances automatically join the registry, and failed instances are removed through health checks. It's the backbone of client-side or server-side load balancing in distributed systems.
2. How it Works in .NET Core
Common implementations in .NET:
  • Consul - with Consul.AspNetCore NuGet package for registration and Consul client for discovery
  • Eureka - using Steeltoe.Discovery.Eureka for Netflix OSS integration
  • Kubernetes DNS - built-in service discovery via DNS names (e.g., http://order-service:8080)
  • Azure Service Fabric - native service registry via Naming Service
  • Service Mesh - Istio/Linkerd handles discovery transparently
For HTTP clients, use HttpClientFactory with dynamic address resolution from the registry.
3. Practical Example
In a multi-tenant SaaS platform I worked on, we used Consul for service discovery across 15+ microservices running in Docker Swarm (before we migrated to Kubernetes). Each service registered itself on startup with health check endpoints. 
// Program.cs - Service Registration with Consul
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton<IConsulClient, ConsulClient>(p => new ConsulClient(cfg =>
{
    cfg.Address = new Uri(builder.Configuration["Consul:Host"]);
}));

var app = builder.Build();

// Register service on startup
var lifetime = app.Services.GetRequiredService<IHostApplicationLifetime>();
var consul = app.Services.GetRequiredService<IConsulClient>();
var serviceName = "payment-service";
var serviceId = $"{serviceName}-{Guid.NewGuid()}";

lifetime.ApplicationStarted.Register(async () =>
{
    var registration = new AgentServiceRegistration
    {
        ID = serviceId,
        Name = serviceName,
        Address = "192.168.1.10",
        Port = 5001,
        Check = new AgentServiceCheck
        {
            HTTP = "http://192.168.1.10:5001/health",
            Interval = TimeSpan.FromSeconds(10),
            Timeout = TimeSpan.FromSeconds(5)
        }
    };

    await consul.Agent.ServiceRegister(registration);
});

lifetime.ApplicationStopping.Register(async () =>
{
    await consul.Agent.ServiceDeregister(serviceId);
});

app.MapGet("/health", () => Results.Ok("Healthy"));
app.Run();

// Consuming service - Discovery and Load Balancing
public class PaymentServiceClient
{
    private readonly IConsulClient _consul;
    private readonly HttpClient _httpClient;

    public PaymentServiceClient(IConsulClient consul, HttpClient httpClient)
    {
        _consul = consul;
        _httpClient = httpClient;
    }

    public async Task<PaymentResult> ProcessPaymentAsync(PaymentRequest request)
    {
        // Query Consul for healthy instances
        var services = await _consul.Health.Service("payment-service", "", true);
        
        if (!services.Response.Any())
            throw new Exception("No healthy payment service instances");

        // Simple random load balancing
        var service = services.Response[Random.Shared.Next(services.Response.Length)];
        var baseUrl = $"http://{service.Service.Address}:{service.Service.Port}";

        var response = await _httpClient.PostAsJsonAsync($"{baseUrl}/api/payments", request);
        return await response.Content.ReadFromJsonAsync<PaymentResult>();
    }
}
We had a bug where services weren't deregistering properly on crashes, causing 503 errors. Fixed it by reducing Consul's health check interval to 10 seconds and implementing proper graceful shutdown.
4. Key Takeaway
Service registries solve the "how do I find other services" problem in dynamic environments. Always implement health checks, handle registry unavailability gracefully, and consider using Kubernetes' built-in DNS for simpler deployments.

Q47: Communication - REST vs gRPC basics
1. Theory
REST (Representational State Transfer) uses HTTP/1.1 with JSON/XML payloads, text-based communication, and standard HTTP verbs (GET, POST, PUT, DELETE). It's human-readable, widely supported, and works well for public APIs. gRPC (Google Remote Procedure Call) uses HTTP/2 with Protocol Buffers (binary serialization), supports bidirectional streaming, and generates strongly-typed client/server code from .proto files. gRPC is significantly faster due to binary serialization and multiplexing but requires HTTP/2 support and isn't browser-friendly without a proxy.
2. How it Works in .NET Core
REST in .NET:
  • Create APIs with WebApplication.CreateBuilder() and Minimal APIs or Controllers
  • Use Results.Ok(), Results.BadRequest() for responses
  • Consume with HttpClient via HttpClientFactory
  • Serialize with System.Text.Json (default in .NET 5+)
gRPC in .NET:
  • Add Grpc.AspNetCore NuGet package
  • Define services in .proto files (Protocol Buffers)
  • Build generates C# service base classes and clients
  • Register with app.MapGrpcService<MyService>()
  • Create clients with GrpcChannel and generated stubs
3. Practical Example
In a real-time trading platform, we used REST for external client APIs (mobile apps, web dashboards) and gRPC for internal service-to-service communication between the OrderEngine, RiskService, and MarketDataService. The performance difference was massive—gRPC reduced latency from ~50ms to ~8ms for high-frequency calls. 
// REST API Example - Order Controller
app.MapPost("/api/orders", async (CreateOrderRequest request, IOrderService orderService) =>
{
    var result = await orderService.CreateOrderAsync(request);
    return Results.Ok(result);
});

// Consuming REST API with HttpClient
var response = await _httpClient.PostAsJsonAsync("/api/orders", orderRequest);
var result = await response.Content.ReadFromJsonAsync<OrderResponse>();

// gRPC Example - order.proto file
syntax = "proto3";

option csharp_namespace = "TradingPlatform.Orders";

service OrderService {
  rpc CreateOrder (CreateOrderRequest) returns (OrderResponse);
  rpc StreamMarketData (SubscribeRequest) returns (stream MarketUpdate);
}

message CreateOrderRequest {
  string symbol = 1;
  int32 quantity = 2;
  double price = 3;
}

message OrderResponse {
  string order_id = 1;
  string status = 2;
}

// gRPC Server Implementation
public class OrderServiceImpl : OrderService.OrderServiceBase
{
    public override async Task<OrderResponse> CreateOrder(
        CreateOrderRequest request, 
        ServerCallContext context)
    {
        // Process order logic
        return new OrderResponse 
        { 
            OrderId = Guid.NewGuid().ToString(), 
            Status = "Pending" 
        };
    }

    public override async Task StreamMarketData(
        SubscribeRequest request,
        IServerStreamWriter<MarketUpdate> responseStream,
        ServerCallContext context)
    {
        while (!context.CancellationToken.IsCancellationRequested)
        {
            await responseStream.WriteAsync(new MarketUpdate { Price = GetLatestPrice() });
            await Task.Delay(100);
        }
    }
}

// Program.cs registration
app.MapGrpcService<OrderServiceImpl>();

// gRPC Client Usage
var channel = GrpcChannel.ForAddress("https://localhost:5001");
var client = new OrderService.OrderServiceClient(channel);

var response = await client.CreateOrderAsync(new CreateOrderRequest 
{ 
    Symbol = "AAPL", 
    Quantity = 100, 
    Price = 150.50 
});

Console.WriteLine($"Order ID: {response.OrderId}");
Key lesson: gRPC streaming was perfect for real-time market data. We pushed 1000+ price updates/second to clients without the overhead of WebSocket handshakes. But debugging gRPC is harder—we used Wireshark and grpcurl for troubleshooting.
4. Key Takeaway
Use REST for public APIs and browser compatibility; use gRPC for high-performance internal service communication, especially when you need streaming or sub-10ms latency. You can run both in the same ASP.NET Core app.

Q48: Resilience - Circuit breaker pattern
1. Theory
The Circuit Breaker pattern prevents an application from repeatedly attempting to execute an operation that's likely to fail, protecting downstream services from cascading failures. It has three states: Closed (normal operation, requests pass through), Open (after threshold failures, requests fail immediately without calling the service), and Half-Open (after timeout, allows limited test requests to check if the service recovered). If test requests succeed, it transitions back to Closed; otherwise, it reopens. This improves system resilience and prevents resource exhaustion from retrying failed operations.
2. How it Works in .NET Core
.NET implements circuit breakers via Polly library (now part of Microsoft.Extensions.Http.Resilience in .NET 8+):
  • Polly - AddPolicyHandler() with CircuitBreakerAsync policy
  • Microsoft.Extensions.Http.Resilience (.NET 8+) - AddStandardResilienceHandler() includes circuit breaker, retry, and timeout
  • Configure failure thresholds, break duration, and sampling duration
  • Combine with retry policies for comprehensive resilience
  • Monitor circuit state via OnBreak, OnReset callbacks
In .NET 10, the AddStandardResilienceHandler is the recommended approach.
3. Practical Example
In a payment processing system, we integrated with a third-party fraud detection API that occasionally had outages. Without circuit breakers, our entire checkout flow would grind to a halt waiting for timeouts. We implemented Polly's circuit breaker to fail fast and fallback to a simplified fraud check. 
// Program.cs - Circuit Breaker with Polly (.NET 10)
using Polly;
using Polly.CircuitBreaker;

var builder = WebApplication.CreateBuilder(args);

// Modern approach with Microsoft.Extensions.Http.Resilience
builder.Services.AddHttpClient<IFraudDetectionClient, FraudDetectionClient>(client =>
{
    client.BaseAddress = new Uri("https://api.fraudcheck.com");
})
.AddStandardResilienceHandler(options =>
{
    // Configure circuit breaker behavior
    options.CircuitBreaker.FailureRatio = 0.5; // Open after 50% failures
    options.CircuitBreaker.SamplingDuration = TimeSpan.FromSeconds(30);
    options.CircuitBreaker.MinimumThroughput = 10; // Need 10 requests before evaluating
    options.CircuitBreaker.BreakDuration = TimeSpan.FromSeconds(60); // Stay open for 60s

    // Retry configuration
    options.AttemptTimeout.Timeout = TimeSpan.FromSeconds(5);
    options.TotalRequestTimeout.Timeout = TimeSpan.FromSeconds(15);
});

// Client Implementation with Fallback Logic
public class FraudDetectionClient : IFraudDetectionClient
{
    private readonly HttpClient _httpClient;
    private readonly ILogger<FraudDetectionClient> _logger;

    public FraudDetectionClient(HttpClient httpClient, ILogger<FraudDetectionClient> logger)
    {
        _httpClient = httpClient;
        _logger = logger;
    }

    public async Task<FraudCheckResult> CheckTransactionAsync(Transaction transaction)
    {
        try
        {
            var response = await _httpClient.PostAsJsonAsync("/api/check", transaction);
            response.EnsureSuccessStatusCode();
            return await response.Content.ReadFromJsonAsync<FraudCheckResult>();
        }
        catch (BrokenCircuitException ex)
        {
            // Circuit is open - fail fast with fallback
            _logger.LogWarning("Circuit breaker open, using fallback fraud check");
            return FallbackFraudCheck(transaction);
        }
        catch (HttpRequestException ex)
        {
            // This will be counted towards circuit breaker threshold
            _logger.LogError(ex, "Fraud API call failed");
            throw;
        }
    }

    private FraudCheckResult FallbackFraudCheck(Transaction transaction)
    {
        // Simple rule-based check as fallback
        var isHighRisk = transaction.Amount > 10000 || 
                        transaction.Country == "RU" || 
                        transaction.IsNewCustomer;

        return new FraudCheckResult 
        { 
            IsFraudulent = isHighRisk, 
            Confidence = 0.5,
            Source = "Fallback"
        };
    }
}
We learned this the hard way: during a Black Friday outage of the fraud API, the circuit breaker saved us. Instead of 30-second checkout timeouts, we degraded gracefully with basic fraud checks and processed orders in under 2 seconds. Revenue impact was minimal compared to complete downtime.
4. Key Takeaway
Circuit breakers prevent cascading failures by failing fast when downstream services are unhealthy. Always combine with fallback strategies and monitoring. In .NET 8+, use AddStandardResilienceHandler for production-ready resilience out of the box.
