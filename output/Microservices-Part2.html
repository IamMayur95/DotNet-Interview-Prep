Q41: Scope of services in Dependency Injection
1. Theory
Service scope determines the lifetime of objects created by the Dependency Injection (DI) container. .NET supports three lifetimes: Transient (created each time they're requested), Scoped (created once per HTTP request/scope), and Singleton (created once for the application lifetime). Choosing the wrong scope can cause memory leaks, thread-safety issues, or unexpected behavior. For example, injecting a scoped DbContext into a singleton service will cause "Cannot access a disposed object" exceptions because the DbContext gets disposed before the singleton releases it.
2. How it Works in .NET Core
Register services in Program.cs using:
  • builder.Services.AddTransient<IService, Service>() - New instance every time
  • builder.Services.AddScoped<IService, Service>() - New instance per HTTP request
  • builder.Services.AddSingleton<IService, Service>() - Single instance for app lifetime
  • DbContext is always scoped by default (AddDbContext)
  • HttpClient via AddHttpClient is managed as singleton but with scoped handlers
  • Use IServiceProvider.CreateScope() for manual scope creation in background services
3. Practical Example
In a logistics tracking system, we had a critical bug where a singleton NotificationService was injecting a scoped AppDbContext. The DbContext got disposed after the first request, causing "ObjectDisposedException" on subsequent notifications. We fixed it by making the NotificationService scoped and creating manual scopes in background workers. 
// Program.cs - Service Registration
var builder = WebApplication.CreateBuilder(args);

// Transient - new instance every injection
builder.Services.AddTransient<IEmailSender, EmailSender>();

// Scoped - one instance per HTTP request (or manually created scope)
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddDbContext<AppDbContext>(options => 
    options.UseNpgsql(builder.Configuration.GetConnectionString("DefaultConnection")));

// Singleton - one instance for entire app lifetime
builder.Services.AddSingleton<ICacheService, RedisCacheService>();
builder.Services.AddSingleton<IConfiguration>(builder.Configuration);

// Background service needs manual scope creation
builder.Services.AddHostedService<OrderProcessingWorker>();

var app = builder.Build();
app.Run();

// CORRECT APPROACH - Background Worker with Manual Scopes
public class OrderProcessingWorker : BackgroundService
{
    private readonly IServiceProvider _serviceProvider;
    private readonly ILogger<OrderProcessingWorker> _logger;

    public OrderProcessingWorker(
        IServiceProvider serviceProvider, 
        ILogger<OrderProcessingWorker> logger)
    {
        _serviceProvider = serviceProvider;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            // Create a new scope for each iteration
            using (var scope = _serviceProvider.CreateScope())
            {
                var orderService = scope.ServiceProvider.GetRequiredService<IOrderService>();
                var dbContext = scope.ServiceProvider.GetRequiredService<AppDbContext>();

                try
                {
                    await orderService.ProcessPendingOrdersAsync();
                    _logger.LogInformation("Processed pending orders");
                }
                catch (Exception ex)
                {
                    _logger.LogError(ex, "Error processing orders");
                }
            } // Scope disposed here - DbContext cleaned up

            await Task.Delay(TimeSpan.FromMinutes(5), stoppingToken);
        }
    }
}

Q42: Health checks of API
1. Theory
Health checks are endpoints that report the operational status of an application and its dependencies (database, cache, external APIs). They're used by orchestrators (Kubernetes, Docker Swarm), load balancers, and monitoring systems to determine if an instance should receive traffic. ASP.NET Core supports two types: liveness (is the app running?) and readiness (is the app ready to handle requests?). Health checks can be basic (just responds 200 OK) or detailed (checks DB connections, disk space, memory). Kubernetes uses liveness probes to restart failing pods and readiness probes to route traffic.
2. How it Works in .NET Core
Use the built-in Microsoft.Extensions.Diagnostics.HealthChecks:
  • builder.Services.AddHealthChecks() - registers health check services
  • app.MapHealthChecks("/health") - exposes health check endpoint
  • AspNetCore.HealthChecks.* packages - pre-built checks for SQL Server, PostgreSQL, Redis, RabbitMQ, Azure services
  • Custom checks implement IHealthCheck interface
  • HealthCheckOptions - customize response format (JSON, plain text)
  • Use tags to separate liveness (/health/live) and readiness (/health/ready)
3. Practical Example
In a high-traffic e-commerce API deployed on AKS, we implemented comprehensive health checks after a production incident where unhealthy pods continued receiving traffic because they responded 200 OK while the database connection was dead. We added database, Redis, and external payment gateway checks with separate liveness/readiness endpoints. 
// Program.cs - Health Check Configuration
using Microsoft.Extensions.Diagnostics.HealthChecks;

var builder = WebApplication.CreateBuilder(args);

// Add health checks with dependencies
builder.Services.AddHealthChecks()
    // Liveness checks - basic app health
    .AddCheck("self", () => HealthCheckResult.Healthy("API is running"), tags: new[] { "live" })
    
    // Readiness checks - dependency health
    .AddNpgSql(
        builder.Configuration.GetConnectionString("DefaultConnection"),
        name: "postgres",
        tags: new[] { "ready", "db" },
        timeout: TimeSpan.FromSeconds(5))
    
    .AddRedis(
        builder.Configuration.GetConnectionString("Redis"),
        name: "redis-cache",
        tags: new[] { "ready", "cache" })
    
    .AddRabbitMQ(
        rabbitConnectionString: builder.Configuration["RabbitMQ:ConnectionString"],
        name: "rabbitmq",
        tags: new[] { "ready", "messaging" })
    
    // Custom health check for external API
    .AddCheck<PaymentGatewayHealthCheck>(
        "payment-gateway", 
        tags: new[] { "ready", "external" });

var app = builder.Build();

// Liveness endpoint - just checks if app is running
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("live"),
    AllowCachingResponses = false
});

// Readiness endpoint - checks dependencies
app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("ready"),
    AllowCachingResponses = false,
    ResponseWriter = WriteDetailedHealthResponse // Custom JSON response
});

// Full health endpoint with detailed status
app.MapHealthChecks("/health", new HealthCheckOptions
{
    ResponseWriter = WriteDetailedHealthResponse
});

app.Run();

// Custom response writer for detailed JSON
static Task WriteDetailedHealthResponse(HttpContext context, HealthReport report)
{
    context.Response.ContentType = "application/json";
    
    var result = JsonSerializer.Serialize(new
    {
        status = report.Status.ToString(),
        duration = report.TotalDuration.TotalMilliseconds,
        checks = report.Entries.Select(e => new
        {
            name = e.Key,
            status = e.Value.Status.ToString(),
            description = e.Value.Description,
            duration = e.Value.Duration.TotalMilliseconds,
            exception = e.Value.Exception?.Message,
            data = e.Value.Data
        })
    }, new JsonSerializerOptions { WriteIndented = true });

    return context.Response.WriteAsync(result);
}

// Custom Health Check Implementation
public class PaymentGatewayHealthCheck : IHealthCheck
{
    private readonly IHttpClientFactory _httpClientFactory;
    private readonly IConfiguration _configuration;

    public PaymentGatewayHealthCheck(IHttpClientFactory httpClientFactory, IConfiguration configuration)
    {
        _httpClientFactory = httpClientFactory;
        _configuration = configuration;
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context, 
        CancellationToken cancellationToken = default)
    {
        try
        {
            var client = _httpClientFactory.CreateClient();
            var gatewayUrl = _configuration["PaymentGateway:HealthUrl"];
            
            var response = await client.GetAsync(gatewayUrl, cancellationToken);

            if (response.IsSuccessStatusCode)
            {
                return HealthCheckResult.Healthy(
                    "Payment gateway is responsive",
                    new Dictionary<string, object>
                    {
                        { "status_code", (int)response.StatusCode },
                        { "response_time_ms", 100 }
                    });
            }

            return HealthCheckResult.Degraded(
                $"Payment gateway returned {response.StatusCode}");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy(
                "Payment gateway is unreachable", 
                ex);
        }
    }
}

Q43: Caching mechanism in API
1. Theory
Caching stores frequently accessed data in fast storage (memory or distributed cache) to reduce database load and improve response times. There are three main types: In-Memory caching (stored locally in app memory, lost on restart), Distributed caching (shared across multiple instances using Redis/SQL Server), and HTTP Response caching (browser/CDN caches responses). Cache invalidation is critical—choose between time-based expiration (TTL), manual invalidation, or cache-aside pattern. For read-heavy APIs, caching can reduce latency from 200ms to under 10ms. The hard parts are cache coherence (keeping cache in sync with DB) and cache stampede (many requests hitting a cold cache simultaneously).
2. How it Works in .NET Core
.NET provides multiple caching options:
  • IMemoryCache - in-memory caching (single server)
  • IDistributedCache - abstraction for Redis, SQL Server, NCache
  • AddStackExchangeRedisCache() - Redis distributed cache
  • [ResponseCache] attribute - HTTP response caching
  • AddOutputCache() (.NET 7+) - modern output caching middleware
  • CacheTagHelper - for Razor page caching
  • HybridCache (.NET 9+) - combines in-memory + distributed with automatic serialization
3. Practical Example
In a product catalog API serving 10K requests/sec, we implemented a multi-layer caching strategy: in-memory cache for hot products (top 100), Redis distributed cache for the rest, and database as fallback. This reduced database queries by 95% and average response time from 180ms to 12ms. 
// Program.cs - Caching Configuration
var builder = WebApplication.CreateBuilder(args);

// 1. In-Memory Cache (for single instance or hot data)
builder.Services.AddMemoryCache();

// 2. Distributed Redis Cache (for multi-instance deployments)
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = builder.Configuration["Redis:ConnectionString"];
    options.InstanceName = "ProductCatalog:";
});

// 3. Output Caching (HTTP response caching) - .NET 7+
builder.Services.AddOutputCache(options =>
{
    options.AddBasePolicy(policy => policy.Expire(TimeSpan.FromMinutes(5)));
    
    // Cache by query parameters
    options.AddPolicy("ProductsByCategory", policy => 
        policy.Expire(TimeSpan.FromMinutes(10))
              .SetVaryByQuery("category", "page"));
});

var app = builder.Build();
app.UseOutputCache();
app.Run();

// Hybrid Caching Service - In-Memory + Distributed
public interface ICacheService
{
    Task<T> GetOrSetAsync<T>(string key, Func<Task<T>> factory, TimeSpan? expiration = null);
    Task RemoveAsync(string key);
}

public class HybridCacheService : ICacheService
{
    private readonly IMemoryCache _memoryCache;
    private readonly IDistributedCache _distributedCache;
    private readonly ILogger<HybridCacheService> _logger;

    public HybridCacheService(
        IMemoryCache memoryCache, 
        IDistributedCache distributedCache,
        ILogger<HybridCacheService> logger)
    {
        _memoryCache = memoryCache;
        _distributedCache = distributedCache;
        _logger = logger;
    }

    public async Task<T> GetOrSetAsync<T>(
        string key, 
        Func<Task<T>> factory, 
        TimeSpan? expiration = null)
    {
        // 1. Try in-memory cache first (fastest)
        if (_memoryCache.TryGetValue(key, out T cachedValue))
        {
            _logger.LogDebug("Cache HIT (Memory): {Key}", key);
            return cachedValue;
        }

        // 2. Try distributed cache (Redis)
        var distributedData = await _distributedCache.GetStringAsync(key);
        if (distributedData != null)
        {
            _logger.LogDebug("Cache HIT (Redis): {Key}", key);
            var value = JsonSerializer.Deserialize<T>(distributedData);
            
            // Promote to in-memory cache
            _memoryCache.Set(key, value, expiration ?? TimeSpan.FromMinutes(5));
            return value;
        }

        // 3. Cache MISS - fetch from source (database)
        _logger.LogDebug("Cache MISS: {Key}", key);
        var result = await factory();

        // Store in both caches
        var serialized = JsonSerializer.Serialize(result);
        var options = new DistributedCacheEntryOptions
        {
            AbsoluteExpirationRelativeToNow = expiration ?? TimeSpan.FromMinutes(30)
        };

        await _distributedCache.SetStringAsync(key, serialized, options);
        _memoryCache.Set(key, result, expiration ?? TimeSpan.FromMinutes(5));

        return result;
    }

    public async Task RemoveAsync(string key)
    {
        _memoryCache.Remove(key);
        await _distributedCache.RemoveAsync(key);
    }
}

    }
}

Q44: Event-driven architecture - Kafka, RabbitMQ
1. Theory
Event-driven architecture uses asynchronous messaging to decouple services. Instead of direct HTTP calls, services publish events to a message broker (Kafka, RabbitMQ, Azure Service Bus), and other services subscribe to events they care about. RabbitMQ is a message broker optimized for complex routing (exchanges, queues, bindings) with guaranteed delivery and message acknowledgments. Kafka is a distributed event streaming platform built for high throughput, event sourcing, and replay capability—messages are stored in ordered logs (topics) and can be replayed. Use RabbitMQ for traditional messaging patterns (pub/sub, request/reply) and Kafka for event streaming, log aggregation, and real-time analytics.
2. How it Works in .NET Core
Popular .NET libraries for messaging:
  • MassTransit - abstraction over RabbitMQ, Azure Service Bus, Amazon SQS (recommended)
  • NServiceBus - enterprise messaging framework (commercial)
  • Confluent.Kafka - official Kafka client for .NET
  • RabbitMQ.Client - native RabbitMQ client
  • Rebus - lightweight service bus for .NET
Key patterns: Pub/Sub, Competing Consumers, Saga/Choreography for distributed transactions.
3. Practical Example
In a fintech payment processing system, we used RabbitMQ for transactional messages (payment processing, retries, dead letter queues) and Kafka for event streaming (audit logs, analytics, fraud detection). When a payment succeeded, we published an event that triggered 5 downstream services: invoice generation, email notification, accounting, analytics, and fraud detection—all running independently without blocking the payment API response. 
// Program.cs - RabbitMQ with MassTransit
using MassTransit;

var builder = WebApplication.CreateBuilder(args);

// Configure MassTransit with RabbitMQ
builder.Services.AddMassTransit(x =>
{
    // Register consumers
    x.AddConsumer<PaymentSuccessConsumer>();
    x.AddConsumer<SendInvoiceConsumer>();

    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host("rabbitmq://localhost", h =>
        {
            h.Username("guest");
            h.Password("guest");
        });

        // Configure retry policy
        cfg.UseMessageRetry(retry => retry.Exponential(
            retryLimit: 5,
            minInterval: TimeSpan.FromSeconds(2),
            maxInterval: TimeSpan.FromMinutes(5),
            intervalDelta: TimeSpan.FromSeconds(5)
        ));

        // Dead letter queue for failed messages
        cfg.ConfigureEndpoints(context);
    });
});

var app = builder.Build();
app.Run();
